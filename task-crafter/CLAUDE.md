# Task Crafter — 任务铸造器 v2.0

## Purpose

把任何输入（自然语言、零散想法、会议记录、模板片段）转化为**可执行、可验收、可派发、可追责**的强任务指令。

核心承诺：**不是填表，是铸造** — 输出不是信息排列，而是带判断、带策略、带边界、带兜底的执行契约。

## When to use this skill

- 用户说"帮我生成一个任务"、"拆任务"、"派活"、"写个任务"。
- 用户粘贴了一段自然语言描述，需要转化为可执行指令。
- 用户填了 `任务铸造模板.md` 但觉得不够硬。
- 用户说"把这些拆成几个子任务"或"并行执行"。
- 用户需要一个任务写入 CLA 追踪系统或本地文件。

## When not to use this skill

- 用户只是随口一提，没有真要执行（闲聊场景）。
- 输入已经是完整可执行的 Skill 指令（如调用 document-translator）。
- 用户要求的是分析/调研/报告（先走 `调研与方案分析_报告模板.md`，产出的建议再转入 task-crafter）。

## User defaults

除非用户显式覆盖：

1. 输出格式为 Markdown，便于阅读和在 Claude Code 中执行。
2. 任务默认写入 `~/DailyWork2/task_draft/` 目录，文件名带时间戳。
3. 旧 CLA / OpenClaw SQLite 只作为历史参考路径，不默认写入；除非用户明确要求，不触碰 `~/.openclaw/`。
4. 信息不完整时，基于合理假设生成可用版本，并标注「待确认项」。
5. 一次只问最关键的问题，不碎片化追问。
6. 任务编号格式：`TASK-YYYYMMDD-NNN`（如 TASK-20260501-001）。

## Output Strengthening Rules（任务强度增强规则）

**生成任务时，不允许只做字段填充。必须额外完成以下判断：**

### 1. 判断任务类型

根据用户输入，强制选择任务类型：

| 类型 | 特征信号 | Process 默认策略 |
|------|----------|-----------------|
| **调研** | "看看"、"了解"、"调研"、"有什么" | 信息收集→筛选→归纳→结论→建议 |
| **分析** | "为什么"、"分析"、"诊断"、"评估" | 数据读取→现象梳理→根因分析→方案对比→推荐 |
| **设计** | "设计"、"方案"、"架构"、"怎么实现" | 需求确认→约束梳理→方案对比→选型→接口/结构定义 |
| **开发** | "实现"、"写"、"添加功能"、"做一个" | 预检查→备份→最小改动→测试→验证→记录 |
| **修复** | "修"、"Bug"、"报错"、"不工作" | 日志分析→复现→定位→修复→回归测试→验证 |
| **部署** | "上线"、"发布"、"部署"、"迁移" | 环境检查→备份→灰度/ staging→正式→验证→监控 |
| **文档** | "写文档"、"说明"、"README"、"手册" | 受众确认→结构定义→内容撰写→示例→校验→归档 |
| **复盘** | "总结"、"回顾"、"复盘"、"教训" | 事实梳理→数据验证→归因→ actionable insight→后续追踪 |
| **Agent协作** | "改Skill"、"加Agent"、"触发条件"、"模板" | 现有能力扫描→目标定义→边界设定→示例输入输出→验证 |

**规则**：
- 如果用户输入包含多个类型特征，按主导类型输出，但标注"本任务涉及多类型，建议拆分"。
- 如果类型判断置信度低于 70%，标注「任务类型待确认」。

### 2. 判断任务粒度

- **如果预计耗时 > 1 天**：必须建议拆分，并输出拆分建议。
- **如果任务包含两个以上不同目标**：必须拆成子任务。
- **如果任务同时包含"调研 + 实现 + 部署"**：必须分阶段，输出阶段依赖图。
- **复杂度评估**：
  - S：<2 小时，单一文件或单一命令
  - M：0.5-1 天，涉及 2-5 个文件或一个模块
  - L：1-3 天，跨模块或需要设计决策
  - XL：>3 天，必须拆分

### 3. 判断执行主体

根据任务特征和用户语境，判断执行主体：

| 主体 | 特征 | 边界强化 |
|------|------|----------|
| **人类** | 需要主观判断、创意、跨部门沟通 | 输出决策点和操作建议，不输出具体命令 |
| **Claude Code** | 代码修改、文件操作、命令执行 | 必须明确：文件边界、命令边界、验证命令、回滚要求 |
| **OpenClaw** | Skill 改造、Agent 配置、多技能编排 | 输出 Skill 修改目标、触发条件、边界规则、示例输入输出 |
| **多 Agent** | 涉及多个子系统、需要并行或串行 | 输出主任务、子任务、依赖关系、交接格式 |

**规则**：
- 默认假设执行主体是 Claude Code（因为用户在 Claude Code 环境中）。
- 如果任务涉及服务器操作、Skill 改造、跨 Agent 派发，自动切换主体。
- 多主体任务必须标注每个子任务的主体。

### 4. 判断风险等级

| 等级 | 特征 | 强制要求 |
|------|------|----------|
| **低** | 文档、草稿、分析、只读查询 | 标准 RIPEN 即可 |
| **中** | 代码修改、配置修改、依赖调整、文件写入 | 必须包含：备份、验证、回滚 |
| **高** | 数据库操作、服务器操作、权限变更、删除、迁移、生产环境 | 必须包含：备份、回滚、验证、停止条件、人工确认点 |

**规则**：
- 风险等级自动判断，用户可覆盖。
- 高风险任务必须包含至少一个人工确认检查点。
- 任何涉及 `rm`、`DROP`、`DELETE`、`UPDATE` 无 WHERE 的操作自动升级为高风险。

### 5. 补充假设

- 所有推断内容必须标注为「假设」。
- 每个关键假设必须说明：**如果假设不成立，执行方应该怎么处理**。
- 假设必须标注置信度（高/中/低）。

### 6. 强化验收

- Expected Output 必须包含：**最低可接受交付**、**标准交付**、**不合格表现**。
- 每个验收项必须有**检查方式**和**通过标准**。
- 不允许只写"完成文档"、"修复代码"、"优化系统"这类空泛结果。
- 验收项必须可自动化检查（命令/脚本）或人工检查（明确标准）。

### 7. 强化 No-go

- 必须包含：**范围边界**、**文件边界**、**工具边界**、**数据边界**、**决策边界**。
- 对 Claude Code 类任务，必须明确：
  - 是否允许改文件？
  - 是否允许安装依赖？
  - 是否允许执行危险命令？
  - 是否允许联网？

### 8. 强化 Handoff

任务完成后必须输出交接信息：
- 做了什么
- 改了哪里（文件路径列表）
- 如何验证
- 遇到什么问题
- 遗留什么风险
- 下一步建议

## Harvest 借鉴增强

> 把 harvest-tool 的六阶段能力压缩为任务铸造的前置增强步骤。
> 核心目标：**让任务定义建立在已被验证的实践之上，而不是拍脑袋。**

### 触发条件

当 task-crafter 判断任务类型为以下之一时，**主动询问**用户是否进行 Harvest：

| 任务类型 | 是否建议 Harvest | 原因 |
|----------|-----------------|------|
| 调研 | 否 | 调研本身就是信息收集 |
| 分析 | 可选 | 如果涉及技术方案对比，建议 Harvest |
| 设计 | **强烈建议** | 先看业界怎么设计类似系统 |
| 开发 | **强烈建议** | 找开源参考实现，避免重复造轮子 |
| 修复 | 否 | 修复是针对性问题 |
| 部署 | 可选 | 如果涉及新架构，建议 Harvest |
| 文档 | 否 | |
| 复盘 | 否 | |
| Agent协作 | **强烈建议** | 看别人的 Skill/Agent 是怎么设计的 |

**规则**：
- 强烈建议时，主动问用户："这个任务涉及 [设计/开发/Agent]，要我先去 GitHub 看看有没有可以参考的开源项目吗？"
- 可选时，只在用户明确说"参考一下别人的做法"时才触发。
- 用户说"不用"时，直接进入标准铸造流程。

### Harvest 流程（压缩版）

#### Stage 1 — Discover（发现）

**输入**：用户的任务描述 + 任务类型

**操作**：
1. 从任务描述中提取 2-3 个技术关键词
2. 用 GitHub API 或 WebSearch 搜索相关项目
3. 返回 5-10 个候选项目（star 数、语言、最近更新时间）

#### Stage 2 — Evaluate（评估）

**操作**：
1. 读取每个候选项目的 README（用 WebFetch）
2. 与用户任务做匹配度评估：
   - 功能匹配度（解决的是不是同一个问题）
   - 技术栈匹配度（是否 Python/Go/JS）
   - 复杂度匹配度（是简单库还是完整系统）
   - 可借鉴度（设计思路是否可直接迁移）

**输出**：候选项目评估表（匹配度打分 + 借鉴点 + 风险）

#### Stage 3 — Select（选择）

**操作**：
1. 推荐 1-2 个最匹配的项目
2. 询问用户："建议深入分析 [项目A] 的 [某特性] 和 [项目B] 的 [某特性]，要继续吗？"
3. 用户确认后进入 Harvest

#### Stage 4 — Harvest（深入抓取）

**操作**：
1. 读取推荐项目的核心源码文件（不是全量，是关键部分）
2. 抓取：目录结构、核心模块、关键函数、配置方式
3. 用 Read / WebFetch 工具获取内容

#### Stage 5 — Analyze（分析）

**操作**：
对抓取内容做结构化分析：
- 项目类型和定位
- 核心技术方案
- 关键设计模式
- 可直接复用的部分
- 需要改动的部分
- 借鉴思路

#### Stage 6 — Consensus（共识文档）

**操作**：
生成结构化共识，直接成为 task-crafter Input 的一部分：

```markdown
## Harvest 共识

### 做什么
{项目要解决的核心问题}

### 从哪里借鉴
| 来源 | 直接用 | 改动用 | 借鉴思路 |
|------|--------|--------|----------|
| | | | |

### 怎么做
{具体实施路径}

### 不确定性
{未验证的假设/风险点}
```

### Harvest 集成到 RIPEN 的位置

Harvest 的输出直接嵌入到标准铸造流程的 **Input** 段中：

```markdown
## I — Input（输入）

### 已确认输入
...

### Harvest 借鉴分析（如进行了 Harvest）
<!-- 这里插入 Stage 6 Consensus -->

### 需要执行方主动读取
...
```

Harvest 发现的不确定性自动进入 **A — Assumptions** 段。

### Harvest 边界

- Harvest 只在任务类型为 设计/开发/Agent协作 时建议触发
- Harvest 不阻塞主流程，用户说"不用"则跳过
- Harvest 最多分析 2 个项目，避免过度发散
- 如果搜索不到相关项目，如实告知，不强凑
- Harvest 输出必须标注来源和不确定性，不替用户做技术选型决策

## Workflow

### Step 1 — 解析输入 + 三重判断

读取用户输入，同时完成三个强制判断：

1. **任务类型判断**（调研/分析/设计/开发/修复/部署/文档/复盘/Agent协作）
2. **执行主体判断**（人类/Claude Code/OpenClaw/多Agent）
3. **风险等级判断**（低/中/高）

将判断结果显式输出在任务头部：

```markdown
**任务类型**：开发 | **执行主体**：Claude Code | **风险等级**：中
```

### Step 2 — Harvest 借鉴扫描（可选）

如果任务类型为 **设计/开发/Agent协作**：

1. **询问用户**："这个任务涉及 [类型]，要我先去 GitHub 看看有没有可以参考的开源项目吗？"
2. 用户确认后，执行 Harvest 六阶段（Discover → Evaluate → Select → Harvest → Analyze → Consensus）
3. 将 Consensus 嵌入 Input 段
4. 用户说"不用"则跳过

如果任务类型为 **调研/分析/修复/部署/文档/复盘**：直接跳过。

### Step 3 — RIPEN 结构化 + 增强层

按任务类型选择对应的 RIPEN 模板，填充并增强：

| 段 | 增强要求 |
|----|----------|
| **R — Reason** | 必须包含：当前现象、问题影响、任务目标、不做后果、优先级理由、任务来源 |
| **I — Input** | 必须三分：已确认 / 需读取 / 缺失待确认；如进行了 Harvest，包含 Consensus；必须列出合理假设及不成立处理 |
| **P — Process** | 必须包含：预检查、执行策略、执行步骤、检查点、失败处理、中间产物；如进行了 Harvest，Process 可直接引用 Harvest 结论 |
| **E — Expected Output** | 必须分层：最低可接受 / 标准 / 优秀；必须有验收标准（检查方式 + 通过标准）；必须有不合格表现 |
| **N — No-go** | 必须五类：范围 / 文件 / 数据 / 工具 / 决策 |
| **A — Assumptions** | 关键假设 + 置信度 + 不成立处理；Harvest 发现的不确定性自动进入 |
| **R — Risks** | 可能性 + 影响 + 应对 + 监控 |
| **Handoff** | 7 项交接清单 |

### Step 4 — 自检

生成任务后，强制检查以下项。任何一项不通过，输出 ⚠️ 并说明问题：

- [ ] 任务类型是否明确？
- [ ] 执行主体是否明确？
- [ ] 风险等级是否合理？
- [ ] Harvest 是否已询问（如适用）？
- [ ] Expected Output 是否包含最低/标准/优秀三层？
- [ ] 验收标准是否有检查方式和通过标准？
- [ ] No-go 是否包含文件边界和工具边界？
- [ ] 高风险任务是否有备份/回滚/停止条件？
- [ ] 关键假设是否有不成立处理方案？
- [ ] Handoff 信息是否完整？

### Step 5 — 交付

| 用户意图 | 操作 |
|----------|------|
| 只看看 / 讨论一下 | 直接在对话中输出任务指令 |
| 保存到本地 | 写入 `~/DailyWork2/task_draft/TASK-YYYYMMDD-NNN_任务名.md` |
| 写入 CLA 追踪 | 默认不写；用户明确要求时才插入 `~/.openclaw/workspace/cla/db/cla.sqlite` 的 `learning_task` 表 |
| 拆成多任务 | 生成多个任务文件，输出任务依赖图 |
| 并行执行 | 生成 `parallel-task` 调度格式 |

## Task Decomposition（任务拆解规则）

当用户要求"拆成子任务"或任务复杂度为 XL 时，按以下原则拆分：

1. **按阶段拆**：调研 → 设计 → 开发 → 测试 → 部署
2. **按模块拆**：前端 / 后端 / 数据库 / 部署
3. **按依赖拆**：有前置依赖的串行，无依赖的并行
4. **按验收点拆**：每个子任务有独立的可验收输出

每个子任务必须包含：
- 父任务编号（如 `TASK-20260501-001`）
- 子任务编号（如 `TASK-20260501-001-A`）
- 自己的任务类型、执行主体、风险等级
- 精简版 RIPEN（Output 必须有）

输出任务依赖图：

```text
TASK-001-A [调研] → TASK-001-B [设计]
                    ↓
              TASK-001-C [开发] → TASK-001-D [测试]
                                    ↓
                              TASK-001-E [部署]
```

## Boundary analysis

### Usage boundary

- **适合**：任务规划、任务拆解、自然语言转结构化指令、多任务并行调度、风险评估、Harvest 借鉴扫描。
- **不适合**：直接执行代码（这是 Claude Code 本身的功能）、纯分析调研（先用调研模板）、无目的的闲聊。

### Input boundary

- 接受中文、英文、混合输入。
- 接受模板片段（用户填了一半的 RIPEN 模板）。
- 接受文件路径（读取文件内容作为输入的一部分）。
- 不接受图片/音频/视频作为任务输入（需要先用其他 Skill 提取文本）。

### Output boundary

- 默认输出 Markdown 格式任务指令。
- 可选输出：SQLite INSERT 语句、纯文本、JSON。
- 不输出可执行代码（除非任务本身就是"写代码"）。

### Tool boundary

- 使用 Read / Write / Edit 文件工具。
- 可选使用 SQLite 查询（如果 CLA 系统存在）。
- Harvest 阶段使用 WebSearch / WebFetch 获取 GitHub 项目信息。
- 不依赖外部网络 API（除 Harvest 阶段的搜索）。

# OpenClaw Skill 变更自动检测与飞书通知

**优先级**：P1 | **复杂度**：M | **预计耗时**：1-2 天
**创建时间**：2026-05-01

## R — Reason（目的）

1. **当前现象**：OpenClaw 目前有 36 个 Skills，分布在 `~/.openclaw/workspace/skills/`，但 Skill 的增删改没有自动通知机制。
2. **问题影响**：用户可能在不知情的情况下调用过期版本，或错过新功能上线。
3. **任务目标**：实现监控机制，检测 `skills/` 和 `blueprints/` 目录的文件变更，并通过飞书通知用户。
4. **不做后果**：继续人工检查 Skill 状态，效率低，容易遗漏。
5. **优先级理由**：属于 OpenClaw 运维基础设施，支撑后续 Skill 自治迭代。
6. **任务来源**：OpenClaw 基础设施优化需求

## P — Process（过程要求）

### 预检查
- [ ] 先读取生产日志，不直接改配置

### 执行策略
先诊断后修复，最小改动，每步验证，随时可回滚。

### 执行步骤
1. **诊断**：读取日志，确认错误时间、频率、触发条件
2. **定位**：检查连接池配置和代码中连接获取/释放逻辑
3. **方案选择**：
   - 方案A：增大连接池（快速缓解）
   - 方案B：修复连接泄漏（根治）
   - 方案C：加连接超时回收（缓冲）
4. **实施**：先备份配置，再修改，再重启/重载
5. **验证**：监控连接数，确认 500 消失
6. **回归测试**：写并发测试，证明修复有效

### 检查点
| 检查点 | 验证方式 | 不通过处理 |
|--------|----------|------------|
| 诊断完成 | 输出根因分析报告 | 继续深挖 |

| 修改后连接数正常 | 监控或命令查询 | 立即回滚 |
| 回归测试通过 | `pytest tests/test_auth_concurrency.py` | 不强制阻塞，但记录 |

### 失败处理
| 场景 | 处理方式 |
|------|----------|
| 增大连接池后仍耗尽 | 说明有连接泄漏，转方案B |

| 发现需要改数据库 | 停止，人工确认 |
| 修复时间超过4小时 | 输出当前进度和临时缓解方案 |
