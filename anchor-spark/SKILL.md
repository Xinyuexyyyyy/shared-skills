# Anchor Spark

## 定位

写文章前的"骨架阶段"工具。

它的工作不是写文章,是把**锚点 → 视角 → 骨架**这一段拆出来,让用户在动笔前先看清楚:

- 我手里有什么真实素材(锚点池)
- 这些素材能从哪几种角度切(6 视角)
- 每种切法的骨架长什么样

## 它在防什么坑

- 凭空想题,跟用户真做过的事脱节
- 只用一种叙事方式(默认"故事四段式"),浪费用户最强的对比/重新分类/逆向/拆穿能力
- 一上来就写正文,跳过"先选切角"这一步
- 切角不带锚点引用,文章变成抽象空话

## 什么时候用

- 用户说"想写点关于 X 的"
- 用户说"我有点想法,但不知道怎么落"
- 用户问"今天能写什么"
- 用户给一个主题/关键词/情绪,需要找出"切哪个角度最爆"
- 用户需要从锚点池找素材,但不知道哪些锚点能挂在主题上

## 不什么时候用

- 用户已经知道写什么、有完整骨架,只要正文 → 走 `sure-content-studio`
- 用户在做评审/审稿 → 走 `review-gate`
- 用户没有任何锚点可用(锚点池为空)→ 引导回 Daily Work / study-research 做小落地,不要硬启发

## 必读资源(按顺序)

1. `~/anchor-pool/INDEX.md` — 当前可用锚点
2. `~/anchor-pool/ANCHOR_TEMPLATE.md` — schema 标准(新 anchors-*.md 都按此格式,解析时按字段读)
3. `~/anchor-pool/seeds/wechat_writing_anchors-2026-05-25.md` — 实证基线(用户 4 篇文章拆解,旧自由格式作为参考)
4. `~/content work/skills/sure-content-studio/references/style-profile.md` — 6 视角 + 真护城河三件套
5. `~/.claude/projects/-Users-sure-Daily-Work/memory/feedback_motivation_model.md` — "灵感来自摩擦"机制(决定何时拒绝启发,引导回做事)

## 工作方式

### Step 1: 接住主题

用户输入可能是:
- 一个名词(`agent` / `自由度` / `躺平`)
- 一个动词短语(`想骂下 prompt 工程师`)
- 一个情绪状态(`今天感觉很被动`)
- 一个具体经历(`刚装完系统`)
- 一个空主题("今天能写什么")

不要让用户先说清楚想写什么——**他要是说得清,就不需要 spark**。

### Step 2: 翻锚点池

跑命令收集所有可用锚点:

```bash
cat ~/anchor-pool/INDEX.md
# 然后按相关性逐个 cat 候选 anchors-*.md
```

**新 anchors-*.md 是结构化的**(见 `~/anchor-pool/ANCHOR_TEMPLATE.md`),解析时按字段读:

```
^## N. <标题>           ← 锚点起点
^- `id`: <YYYYMMDD-WS-NN> ← ID,推荐 spark 时引用此 ID
^- `kind`: <值>          ← kind 必读
^- `face`: <值>          ← 多面标签(可选)
^- `quote`: "<原话>"     ← 引文必读
^- `source`: <出处>      ← 溯源必读
^- `usable_for`: <视角>  ← 推荐视角(可选)
```

匹配规则(按优先级):
1. `quote` 字段直接出现主题关键词
2. `kind` 跟用户表达的紧张/情绪挂钩(用户说"被动" → 优先 `contrast` / `inversion`)
3. `face` 匹配用户当前关心的"面"
4. 锚点产出于**最近 30 天**的(新鲜度优先)
5. `kind=metaphor` 锚点(用户的护城河)优先于普通 `story` 锚点
6. `usable_for` 含目标视角的锚点(用户已暗示能这么用)

挑出 3~5 条,**不要超过 5**。

### Step 3: 6 视角切角

对每个视角,给一个**带锚点引用的骨架**(2~4 行,不写完整段落)。
锚点引用用 `[anchor:<id>]` 形式——既能让用户跳查,也让 stats.sh 之后能检测消费率。

```markdown
### 1. story (我做了 X,发现 Y)
- **骨架**: [开篇用某锚点的具体场景] → [中段揭示 X] → [结尾落到 Y]
- **用到的锚点**: [锚点 A] [锚点 B]
- **本视角强度**: ⭐⭐ (用户用得不多,但这个题适合)

### 2. metaphor (跨学科类比)
- **骨架**: ...
- **用到的锚点**: ...
- **本视角强度**: ⭐⭐⭐⭐⭐ (护城河)

### 3. contrast (大家都说 A 但其实 B)
...

### 4. reclassify (看似 5 类,本质 2 类)
...

### 5. inversion (反过来问)
...

### 6. debunk (拆穿被吹爆的)
...
```

强度评级规则:
- ⭐⭐⭐⭐⭐ = 这个题用这个视角能直接命中用户护城河
- ⭐⭐⭐⭐ = 用户常用这个视角,效果稳
- ⭐⭐⭐ = 适合这个题,但视角有锁
- ⭐⭐ = 能写,但不是最好选择
- ⭐ = 这个题不适合用这个视角(直接说,不要凑)

### Step 4: 推荐 + 拍板权交还

给一句"我会选哪个 + 为什么",然后让用户拍板。

**不要替用户决定**——给信号,不下决定(这是顶层用户行为约束,详见 `feedback_motivation_model.md`)。

### Step 5: 锚点池为空 / 不够用

如果池里没有跟主题相关的锚点(<3 条),**不要硬启发**,直接说:

> 锚点池里没有跟"X"挂得上的素材。
> 这意味着你最近没在这件事上做过摩擦。
>
> 选项:
> A. 换主题(给我看 INDEX 里你有素材的方向)
> B. 回去做一个小落地(具体建议 1~2 个)
> C. 先停,过两天再说

详见 `feedback_motivation_model.md` 的"灵感来自摩擦"机制——这种状态下给写作建议是反作用。

## 输出格式

```markdown
## Spark Output

### 主题: [用户给的主题]

### 找到的相关锚点 (N 条)

1. **[锚点标题]** (`kind: metaphor`) — 来源: anchors-2026-05-26.md
   > "原话引用..."

2. **[锚点标题]** (`kind: contrast`) — 来源: anchors-2026-05-20.md
   > "..."

### 6 视角切角

[6 个视角的骨架,带强度评级]

### 我的推荐

视角 [N],理由 [一句]。

但你拍板。挑哪个?
```

## 用户可见检查点

### 检查点 1: 必须给真锚点引用

每个视角骨架里必须能指到具体哪条锚点。
**禁止凭空给骨架**——那变成普通选题建议,不是 spark。

### 检查点 2: 必须给 6 个视角

不是给"最相关的 3 个"。**6 个全给**,差的视角直接打 ⭐ 评级,但要呈现——这本身就是给用户看"哪些路被堵了"。

### 检查点 3: 锚点池空时不要硬启发

宁可什么都不输出,也不要给"通用写作建议"。

## 边界条件

### 情况 1: 用户给的是空主题("今天能写什么")
- 翻 INDEX 看最近 7~30 天产出最丰富的方向
- 给 2~3 个候选主题,让用户挑
- 挑完再走标准 Step 2~5

### 情况 2: 用户给的主题很大(比如"AI")
- 不要勉强覆盖
- 主动收窄:"AI 这个题太大,锚点池里能挂上的有几个具体角度: [agent 名动词] [机械自由度] [0 代码用户]——挑一个?"

### 情况 3: 锚点池里有相关锚点但用户最近没做事
- 用旧锚点没问题,但提示"这个素材是 N 个月前的,你确定还想说这个吗?"
- 让用户判断是不是自己的当下状态

### 情况 4: 用户说"我想写得更像 XX"
- 不接这种请求(详见 `style-profile.md` 的 Anti-Patterns)
- 改成"那个作者的视角是 contrast,你想试 contrast 视角写自己的素材吗?"

## 与其他 skill 的关系

```
spark (找视角骨架) → sure-content-studio (写正文) → review-gate (审稿)
                                                  ↘ stats (复盘消费率)
```

spark 是写作的第一动作,不是唯一动作。

## 最小验收

这轮至少要做到:

1. 翻过 INDEX 和至少一个 anchors-*.md
2. 给的每个视角骨架都能指到具体锚点
3. 6 个视角全部呈现(差的打 ⭐ 不藏)
4. 推荐有,但用户拍板
5. 锚点池不够时直接说,不硬启发
