# Closeout

## 定位

`closeout` 是任务结束时的轻量收尾层。它的目标不是展示内部日志，而是让用户一眼看懂：

- 这轮结果是什么
- 现在怎么验
- 还有什么没完成或不稳
- 是否有必要写入记忆

默认输出必须贴近人的阅读感受。结构服务于理解，不反过来压住内容。

## 核心原则

1. **先可读，再完整**：默认短收尾；细节按需展开。
2. **先结果，再过程**：不要把工具链、路由、失败 prompt 放在开头。
3. **先用户视角，再技术视角**：优先说“你现在能看到/做什么”。
4. **证据底线不降级**：说“完成/生效/通过”必须有真实验证；没有就标“未验证”。
5. **不借收尾扩任务**：下一步只给和本轮相邻的行动。
6. **不擅自写记忆**：只建议，用户确认后才写。

## 默认输出

默认用 2-4 个短段落，不强制标题，不强制旧 6 段。

推荐形态：

```markdown
结论：……

怎么验：……

还剩/风险：……

记忆建议：……（只有确实有必要时才出现）
```

小任务可以更短：

```markdown
已处理好：……  
你可以通过 …… 验证。……
```

大事、失败、阻塞时才使用显式四段：

```markdown
**结论**
……

**怎么验**
……

**未完成**
……

**下一步**
……
```

## 默认隐藏

以下内容默认不展示，除非用户明确要求、任务失败需要定位，或本轮目标就是调试这些机制：

- 实际路由 / 期望路由
- 工具链完整调用过程
- 失败 prompt 原文
- 失效层、责任层、内部判断链
- 建议最小修复文件列表
- 过长的测试矩阵、执行日志、诊断表
- 记忆字段模板、锚点 schema、索引刷新细节

如果必须提工具或验证，只压成一句可读话：

```markdown
我用新会话烟测确认了三类请求正常，论文检索入口还有一处路由偏差。
```

不要默认写成：

```markdown
工具链: A -> B -> C
失败 prompt:
实际路由:
期望路由:
失效层:
```

## 何时展开细节

满足任一条件才展开：

- 用户说“展示路由 / 展示工具链 / 展示证据 / 展开细节 / 复盘过程”
- 结果失败或部分通过，需要说明原因和影响
- 用户要追责、比方案、查为什么
- 本轮任务本身是在调试 route、hook、skill、workflow
- 后续执行需要用户知道具体文件或具体风险

展开时仍然先给一句人话结论，再给细节。

## 状态处理

### 正常完成

说清结果和最短验收路径。不要主动铺执行过程。

### 部分完成

第一句说“部分完成/部分通过”，然后只解释用户需要知道的影响。内部失败细节默认收起。

### 验证失败

第一句说“未通过/还没完成”，再给失败证据、影响范围、下一步。不要把失败包装成完成。

### 阻塞

第一句说卡在哪里，需要谁提供什么。已尝试动作只列关键 1-3 条。

### 用户叫停

说清已做、未做、恢复入口。不要趁机追加新计划。

## 交付证据

按任务类型给最低证据，证据不足就标“未验证”：

| 类型 | 最低证据 |
|---|---|
| 文件/文档 | 路径 + 改了什么 |
| 代码 | 测试/构建/lint 结果；没跑说明原因 |
| UI | 页面 URL 或截图检查；没跑浏览器说明原因 |
| 远端/部署 | 服务状态、URL、日志或版本号 |
| 调研/分析 | 产物路径、核心结论、来源边界 |
| skill / hook / 路由改造 | 修改文件、备份/回滚点、静态检查或烟测结果 |

证据写法要短。不要把命令输出原样倒给用户，除非用户要看。

## 记忆与经验

默认不展示记忆模板，不自动写记忆。

只有在满足以下任一条件时，才轻提示一句：

- 任务跨会话，后续需要接续
- 本轮产生了可复用经验
- 用户明确要求记录
- 修改了规则、skill、hook、路由等长期机制

推荐提示：

```markdown
这轮有一条可复用经验：closeout 默认隐藏内部诊断，按需展开。需要我写入记忆吗？
```

用户确认后再写入 `timeline` / `current-position` / `lessons` / task 记录。用户沉默或否定就不写。

## Lessons 提炼

只提炼“换个会话还会再踩”的结构性经验。不要把一次性网络、手滑、临时错误写成长期规则。

经验写法：

```markdown
当 X 场景出现 → 默认做 Y；只有 Z 时才展开细节。
```

需要用户确认后才落盘。

## 清退 / 锚点 / ideas

这些都是可选扩展，不属于默认收尾。

只有在用户要求，或本轮明确涉及这些机制时，才展开：

- `current-position` 清退建议
- anchors 锚点扫描
- ideas 回看
- lessons 计数更新
- task artifacts 登记细节

默认最多一句话带过：

```markdown
这轮不需要额外清退或锚点处理。
```

没有必要时连这句也可以不说。

## 最小验收

一次合格的 closeout 至少满足：

1. 第一眼能看懂结果。
2. 有最短验收路径。
3. 已验证和未验证分清楚。
4. 没有默认暴露内部路由/工具链/失败 prompt。
5. 没有把记忆、锚点、清退模板铺到用户面前。
6. 没有借收尾扩新任务。

## 坏输出与好输出

坏输出：

```markdown
工具链: 读取任务/路由/skill 文件 -> 新建 4 个 smoke test 线程 -> 读取线程首轮响应 -> 更新文件 -> rg 验证
失败 prompt:
实际路由:
期望路由:
失效层:
建议最小修复文件:
```

好输出：

```markdown
结论：真实新线程测试是部分通过；普通调研、PRD 前发现和轻量问答正常，但论文检索入口还会绕过统一入口。

怎么验：结果已记录在 `smoke-test.md`；你只需要看“部分通过”的结论和剩余风险。

还剩：下一步要单独分析路由层，重点看学术子 skill 为什么会抢入口。
```

如果用户说“把路由细节展开”，再补实际路由、期望路由、失败 prompt 和修复文件。
