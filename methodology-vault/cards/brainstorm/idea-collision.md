# Idea Collision Brainstorm

## Method Summary

Do not only widen. Collide the user's base idea with adjacent fields, competitor patterns, open-source practices, and method patterns, then narrow to options that improve the original.

## Inputs Needed

- User facts: original idea, desired outcome, audience, constraints, taste.
- Project constraints: MVP size, time, available skills, acceptable risk.
- Evidence needed: adjacent solutions, competitor patterns, method families, open-source examples.

## Steps

1. Define the base idea in one sentence.
2. Generate 3-5 directions from different lanes: direct, adjacent, borrowed method, inversion, and future-system.
3. For at least two directions, write a collision card:
   `Base idea -> Adjacent solution -> Borrowed method -> New option -> Why better/worse -> Evidence needed`.
4. Discard weak options explicitly.
5. Recommend one primary direction and one backup direction.

## Outputs

- Main artifact insertion: recommended direction and discarded directions.
- Appendix / ADR / evidence note: collision cards and evidence needed.

## Evidence / Sources

- Source family: Double Diamond, design-thinking ideation, analogy/cross-domain problem solving, competitive pattern scanning.
- Cross-check: at least one adjacent example or method pattern.

## Adaptation Notes

For light-code users, collisions should produce buildable ideas, not novelty for its own sake.

## Failure Modes

- Directions differ only by wording.
- Brainstorm never narrows.
- Collisions are imaginative but not useful.
- Evidence needs are not named.

## Verification

The brainstorm passes when it creates at least one option the user did not originally ask for, and explains why that option is worth considering or rejecting.
