# 可执行 PRD 减法

## 方法摘要

PRD 不是执行交接模块，也不是想法说明书。把 PRD 写成产品判定文档：读者能判断第一版做什么、不做什么、做到什么样算成功、什么情况下必须停下来问人。

主正文只保留最终决策、用户可见范围、UI 行为、数据位置和验收标准；调研过程、长对比、来源表和被拒方案移到附录 / ADR / evidence brief。

## 需要输入

- 用户事实：目标用户、问题、期望结果、偏好、约束、非目标。
- 项目约束：MVP 边界、时间、工具、数据敏感性、部署目标。
- 证据需求：关键假设、竞品/来源可信度、验证路径。

## 操作步骤

1. 写出最小可判定切片：目标用户、痛点、MVP wedge、非目标、成功信号。
2. 把 MVP 写成用户可见结果，不只写功能名。
3. 补影响实现的细节：页面、动作、按钮、状态、输入、输出、数据字段、存储路径、权限、任务和验收。
4. 核心行为用 `Requirement / Scenario` 表达；关键场景必要时补 `Given / When / Then`。
5. 把方法解释、来源备注、被拒方案和长取舍移到附录、ADR 或 evidence brief。
6. 跑 PRD Gate：`Boundary x Detail x Evidence x Acceptance x Stop Conditions`。
7. 如果硬指标缺失，回滚提问 / 补调研，或降级为 `discovery_note` / `decision_brief`。

## 输出

- 主正文插入：能支撑后续拆任务的简洁 PRD 段落。
- 附录 / ADR / evidence note：方法依据、来源表、长方案对比、被拒方向。

## 证据 / 来源

- 来源族：成熟 PRD 模板、用户故事实践、发布标准、开放问题追踪。
- 交叉检查：按真实下游需要反问，读者是否能判断范围、行为、验收和停下条件？

## 适配说明

对轻代码 MVP，优先少章节、强细节。按钮、数据路径、最短验收路径清楚的短 PRD，优于堆满产品话术的长 PRD。

## 失败模式

- 主 PRD 仍然像调研笔记。
- UI 行为只有名字，没有步骤、按钮和状态。
- 数据位置和目录结构缺失。
- 验收是感觉，不是最短测试路径。
- 风险没有写成 Stop Conditions，导致不该执行时继续执行。

## 验证

问：读者只看这份 PRD，能否判断 V1 做什么、不做什么、怎么操作、怎么验收、什么情况下必须停下来问人？如果不能，先回滚补缺口，不要继续加正文。
