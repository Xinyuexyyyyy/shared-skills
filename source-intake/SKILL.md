# Source Intake — 第一轮材料分析

## 定位

把 URL、网页、PDF、本地文件、粘贴材料或资料包先分析清楚，并直接在 agent 界面向用户说明材料内容。

它的首要目的，是帮用户更快判断一份材料：这是什么、讲什么、结构如何、哪里重要、哪里不可靠、是否值得继续。

它不是正式 research，不是学习地图生成器，不是写作前处理，不是代码实现前处理，也不是重型 OCR 工厂。后面往哪里接，听用户的；用户没指定下一步时，不主动规划学习、调研、写作或实现路线。

## Outcome Contract

- **产出：** agent 界面内的初读报告，或明确说明哪些材料无法读取。
- **完成标准：** 用户能快速判断资料是什么、讲什么、重点在哪、风险在哪、是否值得继续。
- **证据：** URL、文件路径、标题、作者/机构、日期、访问时间、页码/段落位置、短摘录或可靠转述。
- **输出：** 直接在对话里交材料状态、内容判断和未处理风险；不默认另行落 `.md`，不把材料分析包装成最终观点。

## 什么时候用

- 用户给 URL、网页、PDF、DOCX、Markdown、截图文字、长粘贴文本或资料包，说“读一下”“整理一下”“先收进来”。
- 用户想先知道一份材料是什么、讲什么、值不值得看、哪里重要、风险在哪。
- 用户只需要第一轮材料分析，还没决定要不要进入学习、调研、写作、方案或代码实现。
- 用户只需要抽取事实、结构、术语、数据、引用点，还没要求正式综合判断。
- 用户给出较长 PDF、教材、课件、扫描件或图表密集资料，需要先做文字快读，并轻量标注是否需要二阶段视觉复核或服务器处理。

## 不什么时候用

- 用户已经明确要正式调研、竞品分析、论文综述，应走 `research`，必要时先用本 skill 做前置摄取。
- 用户要快速理解陌生概念而不是摄取资料，应走 `fast-onboard`。
- 用户要生成成稿、排版、导出，应走 `output-layer` / `output-polisher`。
- 用户要 UI 页面审查，应走 `ui-design`。
- 用户明确要学习计划、练习或知识卡片，应走学习相关能力；本 skill 只在用户要求先分析材料时使用。
- 用户只给一个主题，没有任何资料或链接，应先走 `idea-to-research` 或 `research`。

## 输入类型

| 输入 | 处理 |
|---|---|
| URL / 网页 | 读取页面内容，记录标题、URL、访问时间、发布时间；不能读取时说明原因 |
| PDF / DOCX / Markdown / TXT | 读取文件，保留文件路径、页码或章节位置；不能打开时说明工具缺失 |
| 粘贴文本 | 记录为 `conversation` 来源，保留用户给出的上下文 |
| 多文件资料包 | 先列清单，再按文件类型分批摄取 |
| 图片或截图 | 只抽取可见文字和版面线索；需要高精度 OCR 时先说明限制 |

## 工作流

### Step 1：确认分析目标

先判断用户要的是哪一种：

- **initial-read**：默认模式；说明材料身份、结构、重点、价值、风险和是否值得继续。
- **evidence-light**：用户明确要证据、引用或来源时，才在初读报告里增加轻量证据点。
- **file-batch**：多文件资料包，先列材料清单，再给总体判断。

如果目标不清，但材料明确，默认走 `initial-read`，不要频繁追问。

### Step 2：建立 source list

先列来源清单：

| id | type | title | source | status | note |
|---|---|---|---|---|---|

规则：

- 每个来源必须有稳定 `id`，如 `S1`、`S2`。
- URL 必须记录完整链接和访问时间。
- 本地文件必须记录绝对路径或用户给出的相对路径。
- 不能读取的来源也要进入 source list，`status=failed`，写明原因。

### Step 3：默认第一遍文字快读

除非用户明确要求精读公式、图表、OCR、截图或服务器处理，否则第一遍只做文字快读。

默认第一遍只做：

- 抽取可用文字层。
- 识别标题、目录、章节、段落结构。
- 提炼核心概念、主线、术语、关键事实和证据点。
- 在 agent 界面直接输出初读报告。
- 用户明确要求落盘、材料很长需要保留、或任务会中断时，才写 `.md` 文件。
- 对 PDF 轻量记录页数、大小、文字层状态和视觉债提示。

默认第一遍不做：

- 不截图。
- 不 OCR。
- 不做公式识别。
- 不逐页视觉复核。
- 不全量表格还原。
- 不上传服务器。
- 不调用外部视觉模型。

如果文字层可读，即使 PDF 很长，也先交付最小可用阅读结果；长文件可以只抽目录、章节开头、关键页样本和用户目标相关页，不强行全量深加工。

### Step 4：处理范围判断

第一遍快读后，只判断是否“值得继续处理”和“继续前需要用户指定目标”。不要自动规划下游 skill。

默认规则：

| 模式 | 条件 | 处理 |
|---|---|---|
| local-fast | `<=50` 页或 `<=30MB`，文字层清楚，低敏资料 | 本地只读，直接在界面输出初读报告 |
| text-first-large | `>=100` 页或 `>=80MB`，但文字层可读 | 本地文字快读或抽样快读，直接在界面输出结构、主线、重点和风险 |
| server-recommended | 扫描版、图片/公式/图表密集且用户要精读，或需要 OCR/视觉模型/公式识别，或预计超过 10 分钟 | 只给二阶段服务器建议；复制本地文件到服务器、调用外部 OCR 或视觉模型前必须确认 |
| detect-first | 文字层、文件类型或隐私边界不确定 | 本地先做只读检测，输出页数、大小、文字覆盖率、视觉债风险，再说明能否继续读 |

规则：

- 能本地快速读文字的资料，不升级成重流程。
- 长文件默认先文字快读；只有用户要精读视觉内容、文字层不可用或本地处理明显不合适时，才建议服务器分段处理。
- 本地文件外发到服务器、外部 OCR、外部视觉模型、公式识别 API 都属于外发/重处理，必须先确认。
- 用户只要求“先读一下”时，默认先做最小可用阅读，不做全量深加工。
- 第一轮结束时只说“是否值得继续”和“继续需要用户指定目标”，不要主动展开学习/调研/写作路线。

### Step 5：PDF 检测和视觉债

处理 PDF、课件、教材、扫描件和技术文档时，必须区分文本摄取和视觉摄取。第一遍只做轻量检测和风险标注，不默认进入视觉复核。

先记录：

| 字段 | 说明 |
|---|---|
| `pdf_type` | `text_layer / scanned / mixed / unknown` |
| `page_count` | 总页数 |
| `file_size` | 文件大小 |
| `text_layer_coverage` | 可抽文本页比例，无法计算则写 `unknown` |
| `visual_debt_status` | `none / detected / reviewed / blocked` |

页面风险类型：

| 类型 | 判断线索 | 默认处理 |
|---|---|---|
| formula-heavy | 等号、分式、上下标、希腊字母、变量密度高 | 标 `needs_visual_review`，未经复核不升高置信度 |
| chart-heavy | 矢量对象多、坐标轴、曲线、图例、趋势描述 | 第一遍只标页码和上下文；二阶段才截图，不能编造数据点 |
| table-heavy | 网格、列标题、数字矩阵、结构化版面 | 标结构化风险，不能把打乱文本当表格 |
| image-heavy | 图片对象多、正文少 | 标图像语义风险 |
| low-text-visual | 文本极少但页面有内容 | 标 `needs_ocr` 或 `needs_visual_review` |
| mixed | 文字、公式、图表混排 | 拆成文本证据、公式证据、图表证据 |

三轨状态：

```yaml
text_track: extracted | partial | failed
formula_track: not_checked | candidates_found | needs_visual_review | reviewed | failed
figure_track: not_checked | screenshots_ready | described | failed
table_track: not_checked | needs_structure_review | extracted | failed
```

规则：

- 不得声称公式、图表、表格、照片已由普通文本抽取完整覆盖。
- 仅来自 PDF text extraction 的公式，默认 `confidence=low`。
- 未经视觉复核的公式、图表、表格证据，`confidence` 不得高于 `medium`。
- 纯图片或扫描 PDF 不直接判死；先标 `needs_ocr`，说明本地能力边界，再给可选处理路线。
- 第一遍不渲染高风险样本页；只有进入二阶段视觉复核时，才按高风险样本页开始。
- 全量截图、OCR、公式识别、外部视觉模型必须先确认。

### Step 6：抽取正文和结构

对每个可读来源，至少抽取：

- 标题 / 作者 / 机构 / 日期
- 目录或段落结构
- 核心事实
- 关键术语
- 数据、数字、定义、方法、结论
- 与用户目标相关的证据点

不要整篇复制。长文只做结构化摘取和短摘录。

### Step 7：轻量证据点

默认不生成完整 evidence table。只有用户明确要“证据、引用、source log、evidence table、调研/写作支撑”时，才生成结构化证据表。

需要生成时，每条 evidence 至少包含：

| id | source_id | claim | support | location | confidence | note |
|---|---|---|---|---|---|---|

规则：

- `claim` 写成可复用事实，不写成空泛摘要。
- `support` 使用简短摘录或转述；长摘录必须改为概括。
- `location` 写页码、章节、段落、时间戳或 URL 锚点；没有位置就写 `unknown`。
- `confidence` 用 `high / medium / low`。
- 证据不足时写 `low`，不要强行补成确定事实。
- 涉及公式、图表、表格、照片的 evidence，`note` 必须写明 `text_extracted / visual_unverified / visually_checked / needs_ocr` 等状态。

### Step 8：在界面输出初读报告

默认主产物是界面内的初读报告。它不是学习地图，也不是正式调研结论；默认不另行写 `.md` 文件。

报告固定包含：

```markdown
# 初读报告

## 1. 资料身份
[source list]

## 2. 一句话判断
[这份资料是什么，适合先怎么理解]

## 3. 内容结构
[章节、模块、段落框架；没有目录时写 inferred]

## 4. 核心内容
[主线、关键概念、重要事实、可复用证据点]

## 5. 可用价值
[适合拿来做什么]

## 6. 不适合做什么
[边界、不能直接支持的用途]

## 7. 风险和缺口
[无法读取、证据薄弱、时间不明、来源可疑、版权限制]

## 8. PDF And Visual Status
[pdf_type、text_layer_coverage、第一遍是否只读文字、视觉债、三轨状态、二阶段建议；非 PDF 可写 not_applicable]

## 9. 是否值得继续
[值得 / 暂不值得 / 需要补材料 / 需要用户指定下一步目标]
```

结尾规则：

- 如果用户没有指定下一步，结尾只写：`如果要继续，请指定下一步目标。`
- 不主动列出要接哪个 skill。
- 不主动生成学习路径、研究方案、写作方案或实现方案。

## 输出位置

默认不落盘，直接在 agent 界面输出。

只有以下情况才写入 `task-draft/source-intake/<slug>/`：

- 用户明确要求保存成 `.md`。
- 材料很长，界面输出会丢失关键结构。
- 长任务可能中断，需要保留阶段结果。
- 用户要求 source log、evidence table 或后续复用。

可选文件：

- `initial-read-report.md`
- `source-log.md`（用户需要来源清单时）
- `evidence-table.md`（用户明确要证据表时）
- `manifest.json`（如需要机器读取）
- `visual-debt-manifest.csv`（进入二阶段视觉债审计时）
- `visual-debt-summary.json`（进入二阶段视觉债审计时）
- `visual-review.md`（做过视觉复核时）
- `visual-pages/`（只保存抽样页或高价值页截图，不默认全量保存）

不要为了“规范”默认落 `.md`；用户当前最需要的是在 agent 界面直接看懂材料。

## 用户确认规则

必须先问：

- 要联网读取付费、登录、私密或受权限限制的材料。
- 要批量处理大量文件，可能耗时或覆盖已有输出目录。
- 要把敏感材料、账号、密钥、商业数据写入长期目录。
- 要执行 OCR、转码、解压、批量移动、删除或覆盖文件。
- 要把本地文件复制到服务器，或让服务器处理用户本地资料。
- 要调用外部 OCR、外部视觉模型、公式识别 API 或任何会上传原文/截图的服务。
- 要从第一遍文字快读升级到二阶段视觉复核、公式识别或全量表格还原。
- 要对大 PDF 做全量截图、全量 OCR 或长时间批处理。
- 要从初读直接升级成正式 research、成稿、学习计划或代码实现。
- 要把初读报告写入长期目录或覆盖已有文件。

不必问：

- 用户已给公开 URL，且任务就是读取/整理该页面。
- 用户已给本地路径，且只做只读提取。
- 输出格式没说时，默认在界面输出 Markdown。
- 证据粒度没说时，默认只抽足以支持初读判断的短证据点。
- PDF 文字层可读，且用户只说“读一下”或“快速读资料”时，默认只做第一遍文字快读。
- 用户没有说明下游用途时，默认只出初读报告，不默认落盘，不默认规划承接。

## 工具规则

- 优先使用项目已有读取工具、标准 CLI、浏览器或解析库。
- 读取 URL 时必须记录来源和访问时间。
- 读取 PDF/DOCX 时优先保留页码或章节位置；工具拿不到页码时必须说明。
- 工具失败时先换一种读取方式；仍失败就保留 failed source，不假装读过。
- 不把一次读取失败升级成结论失败；可以交付部分 intake。
- 本地处理优先只读；不要修改原始文件。
- 长任务不要逐页刷屏；对话只输出阶段摘要，页级日志写入文件。
- 服务器适合长文件和重依赖，但不代表默认上传；第一遍文字快读优先本地只读，外发边界先确认。

## 安全和版权规则

- 不迁移密钥、账号、cookie、私密路径或一次性 token。
- 不无确认上传、发布、同步或外发用户文件。
- 不整篇搬运受版权保护材料；只做摘要、结构化抽取和短摘录。
- 不伪造作者、日期、页码、来源链接或访问结果。
- 不把无法验证的材料写成确定来源。
- 不把原 Skill、远端工具或旧系统里的密钥、账号、私密路径迁移到当前输出。
- 不把公式、图表、表格的未核对内容写成高置信证据。
- 不因纯图片 PDF 或扫描件无法本地 OCR 就假装已读取；必须标明 `needs_ocr`。

## 质量检查清单

- [ ] 每个来源都有 `id`、来源、状态和读取方式。
- [ ] 无法读取的来源已写明失败原因。
- [ ] PDF 已判断 `pdf_type`、页数、文件大小和 `text_layer_coverage`；无法判断时已写明。
- [ ] 已明确本轮是否为第一遍文字快读。
- [ ] 已给出本地文字快读、先检测或二阶段服务器建议。
- [ ] PDF 公式、图表、表格、图片、低文本页已做轻量视觉债提示，或说明不适用。
- [ ] 三轨状态已记录：文本、公式、图表/图片、表格。
- [ ] 已在 agent 界面输出初读报告，且能回答资料身份、一句话判断、内容结构、核心内容、重点位置、可用价值、边界、风险和是否值得继续。
- [ ] 未经用户要求，没有默认另行落 `.md` 文件。
- [ ] 未经用户要求，没有默认生成完整 evidence table。
- [ ] 公式/图表/表格 evidence 未经视觉复核时没有标成 `high`。
- [ ] 摘录不过长，版权风险已控制。
- [ ] 已区分事实、推测和下游待分析问题。
- [ ] 没有主动规划学习、调研、写作或实现路线；如需继续，已请用户指定目标。
- [ ] 没有把资料摄取伪装成最终调研结论。

## 失败处理

| 情况 | 处理 |
|---|---|
| URL 读取失败 | 记录 URL、错误、访问时间；尝试浏览器/备用读取；仍失败则标 `failed` |
| PDF 无法解析 | 尝试备用工具；不能提取时只记录文件元信息和失败原因 |
| PDF 太大 | 先做文字快读或抽样快读；界面交付概要；用户确认后再保存或服务器分段处理 |
| 纯图片或扫描 PDF | 标 `needs_ocr`；如需 OCR/视觉模型，先确认工具、隐私和外发边界 |
| 公式/图表提取不可靠 | 标 `needs_visual_review`；只保留页码、上下文和低置信转写 |
| 长任务中断 | 保留已完成文件、失败页码和日志；不要重头假装完成 |
| 内容太长 | 先抽目录、摘要、关键段落和证据点，不整篇搬运 |
| 多来源互相冲突 | 只标冲突，不在 intake 阶段裁判真伪 |
| 来源日期不明 | 标 `date_unknown`，不要猜日期 |
| 涉及隐私或密钥 | 停止外发和落盘，提醒用户确认处理边界 |

## 最小验收
