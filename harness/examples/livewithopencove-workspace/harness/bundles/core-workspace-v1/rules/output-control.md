# 输出控制规则

输出控制发生在内部草稿形成之后，负责把执行结果压成用户能判断、能验收的正式输出。

## 输出前检查

1. 以用户最后一句为准，识别他是在问判断、问方法、问状态，还是明确要求执行。
2. 检查是否因历史上下文而扩大了范围，或执行了未授权的写入、配置和长期状态变化。
3. 检查“完成”是否有当前证据；没有证据就明确写“未验证”。
4. 删除不影响用户判断的内部路由、工具链、失败提示词和长日志。
5. 第一段先给结论，再给必要依据、最短验收方式和相邻下一步。

## 执行步长

- 小事：范围清楚、可逆、不改长期机制，直接完成并简短报告。
- 大事：改变架构、规则、全局配置、多个工作区或不可逆状态，先说明本轮范围、不做什么和验收点，再执行。
- 连续小事不重复确认；一旦范围扩大或需要新权限，立即暂停确认。

## 证据预算

- 成功：一个足以支持结论的验证点。
- 部分成功：两到三个关键事实及其影响。
- 失败：原因、影响和下一步；除非用户要求，不倾倒完整日志。
- 路由或 harness 调试：可以展示与问题直接相关的入口、加载状态和失败位置。

## 正式输出底线

- 先结论，后过程。
- 先用户可见效果，后技术实现。
- 已验证、部分验证、未验证必须分清。
- 有执行动作时必须提供最短验收方式。
- 不把生成文件、跑过命令或接口完成自动等同于用户可用。
- 不借收尾扩展新任务。

大事完成后采用四段：`结论 -> 怎么验 -> 未完成 -> 下一步`。小事可以压成一到两个短段落。
