# Formal Writing Methodology v1（中文正式写作方法论）

## 定位

本方法论回答：`output-layer` 输出 docx 时，怎样写才算我们想要的“书面、正式、逻辑清楚、结构完整”。

它来自中文源库的第一轮提炼，不来自外文写作指南。它不是模板库，不要求所有文档写成政府公文。它是写作判断标准和修订方法。

## 总原则

正式文章不是“更端着”，而是让读者能快速判断：

1. 这份文件要解决什么问题。
2. 文件对象是谁。
3. 依据从哪里来。
4. 具体内容是什么。
5. 结论、行动或边界是什么。
6. 哪些内容需要人工确认。

一句话：

```text
先交代对象和目的，再给依据和内容，最后落到结论、行动和边界。
```

## 一、结构方法论

### 1. 正式文档的五段骨架

适用：正式报告、申请材料、说明书、项目总结、论文式报告。

```text
对象/目的 -> 背景/问题 -> 核心内容/方法 -> 依据/边界 -> 结论/行动
```

不是每篇都必须用这五个标题，但内容必须能对应到这五类职责。

检查问题：

- 第一屏能否看出这份文档给谁看、解决什么问题？
- 中段是否说明了怎么做、依据是什么？
- 结尾是否告诉读者下一步怎么判断或执行？

常见反模式：

- 开头先铺“时代背景”，迟迟不说对象和目的。
- 中间只有口号和原则，没有方法、步骤或证据。
- 结尾只有“具有重要意义”，没有行动或边界。

修订动作：

- 把对象和目的移到第一段。
- 把背景压缩成问题定义。
- 把原则改成动作、条件、对象和结果。

### 2. 申请类三问结构

来源：NSFC 申请书改版说明、国家重点研发计划申报通知。

适用：基金申请、项目申报、研究计划。

```text
为什么做 -> 做什么 -> 凭什么能做
```

内容要求：

- 为什么做：立项依据、问题价值、现实或学术背景。
- 做什么：研究内容、目标、任务、指标。
- 凭什么能做：研究基础、条件、团队、数据、可行性。

常见反模式：

- 只写意义，不写研究路径。
- 只写目标，不写已有基础。
- 只写技术名词，不写问题和验证方式。
- 指标、任务、预算、附件互相脱节。

修订动作：

- 删除“意义重大”“前景广阔”这类空泛句。
- 前移研究对象、核心问题和研究路径。
- 给每个目标补对应任务或验证方式。
- 涉及预算、附件、合作单位时标记 human-gate。

### 3. 技术说明的问题-方案-效果链

来源：CNIPA 专利申请文件撰写资料、标准编制说明。

适用：技术说明、专利交底、工程方案、研发报告。

```text
技术问题 -> 技术方案 -> 技术效果 -> 边界条件
```

内容要求：

- 问题要具体，不能只是“效率低、体验差”。
- 方案要说明关键组成、步骤或机制。
- 效果要能对应问题，最好有指标、对比或应用条件。
- 边界要说明适用条件、限制或未验证内容。

常见反模式：

- 方案像宣传语，没有技术特征。
- 效果和问题不对应。
- 只有功能列表，没有机制说明。
- 图表没有说明，读者不知道图表证明什么。

修订动作：

- 把“优势”改成“解决了哪个问题”。
- 把“功能”改成“结构/步骤/机制”。
- 给图表补“图/表 + 说明 + 结论”。

### 4. 论文式摘要四要素

来源：中国科技论文在线、ChinaXiv、国家哲学社会科学文献中心、中文期刊投稿须知。

适用：摘要、论文式报告、研究总结。

```text
对象/问题 -> 方法/材料 -> 结果/发现 -> 意义/边界
```

自然科学和工程类偏重方法、数据、结果；社科类偏重问题意识、资料来源、论证路径和观点。

检查问题：

- 研究对象是什么？
- 用什么方法、材料或路径？
- 得到了什么结果或观点？
- 适用边界或意义是什么？

常见反模式：

- 摘要只有背景，没有研究结果。
- 只有观点，没有论证路径。
- 只有方法名，没有对象和材料。
- 只有意义，没有边界。

修订动作：

- 第一句明确对象和问题。
- 第二句交代方法、材料或路径。
- 第三句给结果、发现或核心观点。
- 最后一句说明意义、应用范围或局限。

### 5. 说明书任务链

来源：GB/T 19678 使用说明编制、政务服务材料要求。

适用：说明书、操作指南、流程文档。

```text
适用对象 -> 前置条件 -> 操作步骤 -> 注意事项 -> 异常处理
```

常见反模式：

- 只有步骤，没有前置条件。
- 只有正常流程，没有失败处理。
- 注意事项和步骤混在一起。
- 用户不知道什么时候该停止、回退或求助。

修订动作：

- 在步骤前补适用对象和前置条件。
- 给关键步骤补失败情况和处理方式。
- 把注意事项从正文中抽出来单列。

## 二、逻辑方法论

### 1. 每个结论都要有支撑

正式文档可以有判断，但判断必须能追踪到：

- 数据。
- 事实。
- 政策或标准。
- 研究基础。
- 经验观察。
- 推理过程。
- 明确的假设条件。

反模式：

- “效果显著”但没有对比。
- “具备可行性”但没有基础。
- “具有重要意义”但没有对象和影响范围。
- “建议推进”但没有条件和风险。

修订动作：

- 补依据。
- 降级为“待验证判断”。
- 删除无法支撑的结论。
- 把结论拆成“已知事实 + 推断 + 边界”。

### 2. 行动项要有对象、动作、条件、结果

来源：政府通知、工作报告、项目申报通知。

合格行动项：

```text
面向 [对象]，在 [条件/范围] 下，执行 [动作]，达到 [结果/目标]。
```

反模式：

- 加强管理。
- 提升能力。
- 完善机制。
- 推动落实。

这些词不是不能用，而是不能单独用。它们后面必须接对象、动作、条件或结果。

修订动作：

- 问“谁来做？”
- 问“做什么？”
- 问“做到什么程度？”
- 问“按什么依据判断完成？”

### 3. 章节必须服务同一个主目标

每个一级标题都应该回答主目标的一部分。

检查问题：

- 这个章节是否回答文档主题？
- 它和前后章节是什么关系？
- 是背景、方法、结果、讨论、行动，还是附件？

反模式：

- 有一个章节只是堆材料。
- 标题看似正式，但正文偏题。
- 一级标题之间没有递进关系。

修订动作：

- 合并重复章节。
- 删除游离章节。
- 给章节加一句职责说明。

## 三、内容方法论

### 1. 信息密度优先

正式文章不是越长越好。一个段落至少要承担一种功能：

- 定义对象。
- 说明背景。
- 给出依据。
- 描述方法。
- 呈现结果。
- 说明边界。
- 提出行动。

如果一段话只是在说“重要、必要、积极、显著”，没有对象、依据、动作或结果，应删除或重写。

### 2. 类型决定必要要素

| 文档类型 | 必要要素 |
|---|---|
| 正式报告 | 目的、背景、核心内容、依据、结论/建议 |
| 申请书 | 立项依据、研究内容、研究基础、可行性、预期成果 |
| 科技项目书 | 目标、任务、指标、技术路线、预算/附件边界 |
| 专利/技术说明 | 技术问题、技术方案、技术效果、附图说明 |
| 论文式报告 | 问题、方法、结果、讨论、参考资料 |
| 摘要 | 对象/问题、方法/材料、结果/发现、意义/边界 |
| 说明书 | 适用对象、前置条件、步骤、注意事项、异常处理 |
| 政策/制度 | 适用范围、定义、职责、流程、例外、执行机制 |

### 3. 参考资料是正式性的组成部分

当正文出现多个标准号、链接、文献、政策、指南或数据来源时，应有“参考资料 / 资料来源 / 参考文献”区。

没有来源区的问题：

- 读者无法复核。
- 结论看起来像凭空判断。
- 后续审阅无法定位依据。

## 四、用词方法论

### 1. 正式不是堆大词

慎用：

- 显著提升。
- 全面赋能。
- 革命性。
- 具有重要意义。
- 深度融合。
- 打造闭环。
- 推动高质量发展。

这些词可以出现，但必须被具体对象、动作、指标或范围支撑。

更好的写法：

- “将处理时间从 A 缩短到 B。”
- “用于检查标题层级、参考资料区和图表说明。”
- “适用于申请书初稿，不替代合规审查。”

### 2. 少用低信息连接词

慎用：

- 值得注意的是。
- 需要注意的是。
- 综上所述。
- 从某种意义上说。
- 不可否认的是。

替代方式：

- 直接给结论。
- 直接说边界。
- 直接说明下一步。

### 3. 主体称谓要统一

同一文档中不要混用：

- 我。
- 我们。
- 本人。
- 本项目。
- 该项目。
- 本报告。
- 本文。

推荐：

- 报告类：本报告。
- 项目类：本项目。
- 论文类：本文。
- 申请类：申请人 / 本项目。
- 制度类：本办法 / 本规定。

### 4. 动词要能落地

弱动词：

- 推动。
- 加强。
- 完善。
- 提升。
- 促进。

落地写法：

- 建立。
- 调整。
- 删除。
- 增加。
- 比较。
- 验证。
- 输出。
- 提交。
- 归档。

如果必须用“推动/加强/完善”，后面要补动作对象和完成标准。

## 五、Output Layer 写作棘轮

### Ratchet 1：从来源到 lesson

只有当一条方法得到中文源库支撑，才能进入 lesson。

lesson 格式：

```yaml
lesson_id: FWM-001
name: 申请类三问结构
source_ids:
  - CN-SRC-004
  - CN-SRC-006
applies_to:
  - proposal
condition: 当文档是申请类材料
principle: 先说明为什么做、做什么、凭什么能做
anti_pattern: 只写意义，不写研究内容、可行性或基础
check_question: 是否能找到立项依据、研究内容和研究基础
fix_action: 把研究对象、路径和基础前移，删除空泛意义铺垫
automation: assisted
```

### Ratchet 2：从 lesson 到规则

lesson 可以转成三类规则：

- deterministic：能脚本化。
- assisted：需要模型判断，但必须输出证据。
- human-gate：必须人工确认。

不能直接从 lesson 跳到自动改写。

### Ratchet 3：从规则到修订建议

修订建议必须是动作，不是形容词。

合格：

- “补充研究基础章节。”
- “把结论前移到第一段。”
- “给表格补表题。”
- “把‘意义重大’改为问题、对象和影响范围。”

不合格：

- “更正式。”
- “加强逻辑。”
- “提升表达。”

### Ratchet 4：从真实输出回流

每次 output-layer 处理真实文档后，应记录：

```yaml
case_id: output-docx-20260622-001
doc_type: proposal
triggered_rules:
  - missing_required_section
  - unsupported_conclusion
manual_decision:
  - 用户确认补研究基础
lesson_update:
  - FWM-001 helpful +1
```

如果规则反复误判，应降级或修改。

## 六、第一批 lessons

### FWM-001：申请类三问结构

- **来源：** CN-SRC-004, CN-SRC-006
- **条件：** 当文档是申请书、项目书或研究计划。
- **原则：** 先说明为什么做、做什么、凭什么能做。
- **反模式：** 只写背景和意义，不写研究路径、可行性或基础。
- **检查：** 是否有立项依据、研究内容、研究基础/可行性。
- **修订：** 删除套路化铺垫，把研究对象、路径和基础前移。
- **规则类型：** assisted + human-gate。

### FWM-002：技术说明的问题-方案-效果链

- **来源：** CN-SRC-007, CN-SRC-009
- **条件：** 当文档是技术方案、专利交底、研发报告。
- **原则：** 技术说明必须回答问题、方案、效果和边界。
- **反模式：** 只有功能和优势，没有技术问题和效果依据。
- **检查：** 是否能找到技术问题、技术方案、技术效果。
- **修订：** 把宣传性描述改成问题-方案-效果链。
- **规则类型：** assisted。

### FWM-003：摘要四要素

- **来源：** CN-SRC-010, CN-SRC-012, CN-SRC-013, CN-SRC-014, CN-SRC-015
- **条件：** 当文档是摘要、论文式报告、研究总结。
- **原则：** 摘要至少交代对象/问题、方法/材料、结果/发现、意义/边界。
- **反模式：** 摘要只有背景和意义，没有结果或方法。
- **检查：** 是否缺对象、方法、结果、意义或边界。
- **修订：** 按四要素重排摘要，不新增事实。
- **规则类型：** assisted。

### FWM-004：行动项四件套

- **来源：** CN-SRC-002, CN-SRC-003, CN-SRC-006
- **条件：** 当文档提出建议、措施、任务或执行要求。
- **原则：** 行动项要有对象、动作、条件和结果。
- **反模式：** 只写加强、推动、完善、提升。
- **检查：** 是否能回答谁做、做什么、做到什么程度。
- **修订：** 给每条行动项补对象、动作和完成标准。
- **规则类型：** assisted。

### FWM-005：说明书任务链

- **来源：** CN-SRC-008, CN-SRC-011
- **条件：** 当文档是说明书、操作指南、办理材料。
- **原则：** 说明书要有适用对象、前置条件、步骤、注意事项和异常处理。
- **反模式：** 只有正常步骤，没有失败处理或材料边界。
- **检查：** 是否缺前置条件、异常处理或材料清单。
- **修订：** 把注意事项和异常处理单列。
- **规则类型：** deterministic + assisted + human-gate。

## 七、应用方式

### 写作前

先判断 doc_type：

- proposal
- formal_report
- technical_disclosure
- paper_like
- abstract
- manual
- policy

再选择对应方法论。

### 检查时

先跑 deterministic，再跑 assisted，最后列 human-gate。

### 修订时

只做三类修改：

- 结构重排。
- 表达压缩。
- 缺项提示。

不做：

- 新增事实。
- 代写依据。
- 自动判断合规。
- 模仿政府或期刊口吻。
