# Docx 输出质量规范层 v1

## 定位

`docx-quality-layer` 是 `output-layer` 的质量门，位于风格校正之后、docx 渲染之前。

它不负责写新内容，不替用户补事实，不把排版完成等同于交付完成。它只判断一份准备输出为 docx 的 Markdown 是否足够正式、清楚、完整、可渲染。

## 适用范围

适用：

- 正式报告、说明书、申请材料、研究摘要、论文式文档、项目总结、制度/规范类文档。
- 已有正文基本成型，需要进入 docx 成品化。
- 用户明确要求“正式、书面、逻辑清楚、结构完整”。

不适用：

- 口语稿、公众号表达、访谈稿、创意文案。
- 事实还没确认、材料还没收齐的早期草稿。
- 需要重新研究、重新论证或重写主张的任务。

## 核心质量维度

### 1. 语体正式性

目标：读起来像正式文件，而不是聊天、营销稿或 AI 拼接稿。

检查项：

- 句式克制，少用“非常重要”“革命性”“彻底解决”“赋能”等泛化夸张词。
- 不使用 emoji、口语感叹、随意括号吐槽、标题党式表达。
- 不使用“本文将从多个维度”“值得注意的是”“综上所述可知”等低信息模板句，除非它们承担真实结构功能。
- 术语前后一致；首次出现的缩写、专名、英文名有必要说明。
- 语气不替读者做情绪判断，只呈现事实、判断、依据和行动。

阻断条件：

- 出现明显营销化、口语化或 AI 模板化表达，且影响正式交付。
- 同一概念在正文中用多个名称，导致对象不清。

### 2. 逻辑性

目标：结论能被正文支撑，段落之间有可追踪的推理关系。

检查项：

- 开头明确说明目的、对象或核心结论。
- 每个一级章节服务于同一个主目标，没有游离段落。
- 重要判断有依据、条件或适用边界。
- 因果、递进、转折关系清楚，不用连接词掩盖论证缺口。
- 前后数字、对象、时间、范围不矛盾。

阻断条件：

- 有结论但无依据。
- 前文定义和后文使用不一致。
- 关键行动建议缺少对象、条件或下一步。

### 3. 结构性

目标：文档层级稳定，读者能快速定位信息。

检查项：

- 只有一个总标题。
- 一级标题承担主结构，不把段落标题写成口号。
- 二级标题服务于一级标题，不出现孤立的跳层标题。
- 列表用于并列信息，表格用于比较信息，正文用于解释信息。
- 摘要、背景、问题、方案、结论、附件等模块职责不混用。

阻断条件：

- 标题层级混乱，docx 渲染后无法看出结构。
- 大量列表堆叠但没有解释或结论。
- 章节标题和正文内容不匹配。

### 4. 完整性

目标：按文档类型具备必要要素，不让读者读完还不知道“是什么、为什么、怎么办”。

通用必备：

- 文档目的或交付对象。
- 背景或问题定义。
- 核心内容或方案。
- 依据、边界或约束。
- 结论、行动项或使用方式。

按类型补充：

- **摘要**：研究/项目对象、方法或路径、核心结果、意义或应用边界。
- **说明书**：适用对象、前置条件、步骤、注意事项、异常处理。
- **基金/项目申请类**：立项依据、研究内容/目标、创新点、可行性、基础条件、预期成果。
- **论文式报告**：问题、相关工作或背景、方法、结果、讨论、局限。
- **制度/规范类**：适用范围、定义、职责、流程、例外、执行与更新机制。

阻断条件：

- 缺少文档目的或读者对象。
- 只有原则，没有执行路径。
- 只有现象描述，没有结论或下一步。

### 5. Docx 可交付性

目标：渲染为 docx 后仍是可读、可审、可继续编辑的正式文件。

检查项：

- Markdown 控制符不泄漏到正文，例如裸 `#`、`**`、HTML 片段、未闭合代码块。
- 图片、表格、链接、脚注、引用有明确标题或说明。
- 表格不过宽；单元格内容不塞长段落。
- 引用和参考资料有来源线索，不把裸 URL 当正文。
- 标题、列表、代码块、引用块与正文样式可区分。

阻断条件：

- docx 打开后有明显 Markdown 残留。
- 标题、正文、列表、表格样式无法区分。
- 关键图表或附件缺失说明。

## 严重级别

### Blocker

必须修完才能渲染为正式 docx。

典型情况：

- 目的、对象、结论缺失。
- 结构混乱到无法审阅。
- 事实、数字、对象前后矛盾。
- Markdown 或 HTML 残留影响阅读。

### Warning

可以渲染草稿，但不能作为正式版交付。

典型情况：

- 表达偏口语或模板化。
- 章节职责不够清楚。
- 部分判断缺少依据。
- 表格、列表可读性较差。

### Suggestion

不阻断交付，但建议优化。

典型情况：

- 个别句子偏长。
- 术语说明可以更紧。
- 局部标题可以更准确。

## 检查流程

1. **Preflight**：识别文档类型、目标读者、交付用途；没有这些信息时只允许出草稿。
2. **Rule Check**：跑确定性规则，输出问题清单。
3. **Assisted Review**：对逻辑性、完整性做模型辅助复核，但必须保留问题证据。
4. **Correction**：只修表达和结构，不新增事实；需要新增事实时标为待用户补充。
5. **Render Gate**：Blocker 清零后进入 docx 渲染。
6. **Postflight**：打开或解析 docx，确认样式、标题、列表、表格没有明显异常。

## 输出报告格式

每次质量检查至少输出：

```json
{
  "status": "pass | draft_only | blocked",
  "doc_type": "formal-report | manual | proposal | paper-like | policy",
  "summary": "一句话说明整体质量",
  "blockers": [],
  "warnings": [],
  "suggestions": [],
  "render_notes": [],
  "next_action": "render_docx | revise_text | ask_user"
}
```

每条问题建议使用同一结构：

```json
{
  "dimension": "formality | logic | structure | completeness | docx_readiness",
  "severity": "blocker | warning | suggestion",
  "location": "章节或段落线索",
  "issue": "问题是什么",
  "evidence": "可复核的短证据",
  "fix": "建议怎么改"
}
```

## 第一版规则清单

优先落地 deterministic 规则：

- 总标题数量不是 1。
- 标题层级跳跃，例如 `#` 后直接 `###`。
- 空标题、连续标题、标题后无正文。
- 正文存在 emoji 或明显装饰符。
- 存在未闭合代码块。
- 存在裸 HTML 标签。
- 存在 Markdown 强调符残留风险。
- 句长连续过长。
- 同一模板词重复出现。
- 同一文档混用“我们/本人/本项目/该项目”等主体称谓。

第二批 assisted 规则：

- 章节之间是否有明确递进。
- 结论是否有依据支撑。
- 是否缺少读者完成任务所需的信息。
- 申请类文档是否只写意义、不写可行性。
- 说明书是否只有步骤、没有异常处理。

## 与现有 output-layer 的关系

现有链路：

```text
source.md -> style correction -> clean/obsidian/docx
```

建议链路：

```text
source.md
  -> style correction
  -> docx-quality-layer
  -> corrected.md + quality_report.json
  -> clean/obsidian/docx
```

第一版可以只把 `docx-quality-layer` 当作人工规范和 YAML 预设使用；后续再实现为脚本。
