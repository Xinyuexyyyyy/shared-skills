# 风格适配层规范

## 定位

风格适配是 output-layer 的前置环节。它不是规则匹配替换，而是 LLM 级别的“读懂内容 → 按目标风格重写/调整”。

## 工作流

原始内容 → 风格适配（本环节）→ 用户确认 → 格式输出（现有能力）

## 固定动作（每次都做，不需要用户指定）

1. 去 AI 黑话：删除“需要注意的是”“值得注意的是”“从某种意义上说”“归根结底”等套话
2. 去 AI 逻辑套路：减少“不是…而是…”“与其说…不如说…”等对比句式的过度使用
3. 拆长段：每段只讲一个核心论点，控制在 150 字以内
4. 补过渡：段落之间有自然衔接，不是证据的机械罗列
5. 补判断：关键位置要有作者的明确判断，不能只描述现象

## 场景适配（根据目标场景额外做）

| 场景 | 额外动作 |
|---|---|
| 学术提交 | 引用格式规范、术语一致性、段落逻辑链完整、语气正式 |
| 汇报分享 | 降低术语密度、加“所以呢”、加可视化建议、语气直接 |
| Obsidian 笔记 | 加 frontmatter、加 tag、加 wiki link、结构扁平化 |
| 自媒体发布 | 用用户文风（参考 content work 工作区的文风样本）、口语化、有态度 |

## 触发方式

**默认触发，不需要用户主动调用。**

所有产出文档型内容的 skill 结束后，自动进入风格适配环节：
- survey-writer 写完综述 → 自动进入
- academic-deep-research 产出报告 → 自动进入
- paper-reading 产出 reading_summary → 自动进入

用户可以说”跳过输出层”来跳过，但默认是执行。

## 交互模式

1. 上游 skill 完成后自动进入（或用户说”过一遍输出层”手动触发）
2. 风格适配产出第一版
3. 用户反馈：
   - 具体指出问题 → 针对性修改
   - 给方向（”再直接一点””学术味重一点”）→ 按方向调整
   - 说”不行” → 重跑整体风格
4. 最多 2 轮，第 2 轮还不行就是 prompt/规则需要调整
5. 用户确认通过后，进入自进化环节

## 自进化机制

### 触发条件

- 用户说了”不行”导致重跑 → **必须**问”这次有什么教训值得记住？”
- 用户做了具体修改指令 → **建议**问
- 用户直接说”可以”通过 → 不问

### 教训记录格式

记入工作区的 lessons.md，沿用已有 schema：

```markdown
## YYYY-MM-DD: [标题]
**场景：** [什么类型的内容，什么输出场景]
**观察：** [第一版哪里不对]
**做法：** [怎么改的]
**效果：** [改完好不好]
**置信度：** 高 / 中 / 低
**建议：** [下次遇到类似情况怎么做]
```

### 教训如何回流

风格适配执行前，读取 lessons.md 中与当前场景相关的条目，作为适配 prompt 的一部分。例如：

> 历史教训（自动加载）：
> - 学术综述不要用”不是…而是…”超过 2 次（2026-05-21，置信度高）
> - 汇报文档开头不要用问句，直接给结论（2026-05-22，置信度中）

这样每次风格适配都会”记住”之前踩过的坑，逐步变得更稳定。

## 质量底线（第一版必须达到）

- 全文无明显 AI 腔（套话、黑话、过度对比句式）
- 段落长度合理（无 200+ 字的大段）
- 有作者判断（不是纯描述）
- 结构清晰（该有小标题的有小标题，该有表格的有表格）

## 与现有 style-correction 的关系

- 现有 style-correction（规则匹配）保留，作为风格适配的“最后一道检查”
- 风格适配是 LLM 做的整体调整，style-correction 是规则做的细节兜底
- 两者不冲突，顺序是：风格适配 → style-correction → 格式输出
