# Formal Expansion Rules v1

## 定位

本文件定义正式扩写层的可执行规则。它把“写得更正式、更完整”拆成具体动作，避免扩写变成拉长废话或编造事实。

## 总规则

```text
只扩结构、表达、过渡、边界和行动；不扩事实、数据、依据、结果和合规结论。
```

## 扩写动作

### FE-R001：短输入先建骨架

- **适用：** 原文少于 300 字，且缺少正式标题或章节。
- **动作：** 根据 `doc_type` 建立对应结构骨架。
- **不得做：** 不补不存在的事实、数据、来源。
- **验收：** 扩写稿至少包含对象/目的、问题、处理原则、下一步。

### FE-R002：口语转正式说明

- **适用：** 出现“这个东西、挺、大概、弄一下、很烦、差不多、别乱来”等表达。
- **动作：** 改成对象明确、动作明确、语气克制的说明文。
- **不得做：** 不改变用户原始立场。
- **验收：** 口语词减少，主体和动作更明确。

### FE-R003：重复观点合并

- **适用：** 同一段或相邻段反复出现“提升质量、优化体验、形成闭环、加强管理”等低信息表达。
- **动作：** 保留一次核心判断，并补充对象、动作、条件或边界。
- **不得做：** 不把重复口号扩成更多口号。
- **验收：** 重复表达减少，信息密度提高。

### FE-R004：缺事实时标待补充

- **适用：** 需要事实、数据、政策、案例、项目基础、研究结果才能支撑的内容。
- **动作：** 写成 `待补充：...`，并记录到 expansion report。
- **不得做：** 不编造数字、机构、标准、文献、案例。
- **验收：** 所有事实缺口可在报告中追踪。

### FE-R005：结论必须降级到已有依据范围内

- **适用：** 原文只有意图，没有证据。
- **动作：** 把“已经证明/显著提升/完全解决”降级为“目标是/用于/建议先验证”。
- **不得做：** 不输出超过原文证据范围的确定性结论。
- **验收：** 结论与原文证据强度匹配。

### FE-R006：行动项补四件套

- **适用：** 原文含“加强、推动、完善、提升”等弱行动项。
- **动作：** 尽量补对象、动作、条件、结果；缺信息时标 `待补充`。
- **不得做：** 不指定原文没有给出的责任人或完成时间。
- **验收：** 行动项能回答“谁/对什么/做什么/达到什么结果”中的至少三项。

### FE-R007：保留人工确认边界

- **适用：** 涉及预算、附件、政策、标准编号、版本号、端口、PID、实验结果、申报资格。
- **动作：** 放入“依据与边界”或“需人工确认”段落。
- **不得做：** 不给合规、真实性或技术正确性结论。
- **验收：** 人工确认项在扩写报告和后续 quality gate 中可见。

## 模式差异

| 规则 | conservative | structured |
|---|---|---|
| 建骨架 | 只在原文已有明显结构时微调 | 主动补完整骨架 |
| 改口语 | 是 | 是 |
| 去重复 | 是 | 是 |
| 标待补充 | 是 | 是 |
| 新增章节 | 谨慎，原则上不新增大章节 | 按 doc_type 新增必要章节 |
| 适用场景 | 已有草稿但不正式 | 只有几句话或会议口述 |

## 禁止输出

扩写层不得生成以下内容：

- “根据相关政策规定”但没有政策名称。
- “数据显示”但没有数据来源。
- “实践证明”但没有案例或验证过程。
- “显著提升”但没有对比指标。
- “符合申报要求”但没有当年指南和项目类型。
- “具备创新性/先进性/合规性”这类需要专家或机构判断的结论。

## 扩写报告动作码

| code | 含义 |
|---|---|
| `build_formal_skeleton` | 补正式文档骨架 |
| `formalize_colloquial_expression` | 口语转正式表达 |
| `merge_repeated_claims` | 合并重复观点 |
| `add_boundary_placeholders` | 标注待补充和边界 |
| `downgrade_unsupported_claim` | 降级无依据结论 |
| `complete_action_item_shape` | 补行动项结构 |
| `mark_human_confirmation` | 标人工确认项 |

## 与质量门关系

扩写层只负责生成更好的初稿，不负责判定可交付。

扩写后必须继续跑：

1. `docx_quality_report`：检查稳定结构和渲染前问题。
2. `assisted_quality_report`：检查正式写作方法论缺项。

如果扩写后 `Delivery Recommendation` 仍为 `needs_content_revision`、`needs_input` 或 `blocked`，不得作为正式 docx 交付，除非用户明确选择草稿输出。
