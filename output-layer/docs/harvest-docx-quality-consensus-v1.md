# Docx 输出质量规范层调研共识 v1

## 本轮目标

为 `output-layer` 增加第一版面向 `docx` 的输出质量规范层。它先解决三个问题：

- 把生成内容收束为书面、正式、可交付的表达。
- 检查逻辑性、结构性、完整性，避免“排版出来了但内容不像正式文件”。
- 建立可持续扩展的质量源库，不直接搬运公开文档全文，只保存来源、观察维度和可提炼规则。

本轮不做完整自动评分引擎，不引入 Vale/textlint 运行时依赖，不改变现有 docx 渲染链路。

## 开源项目调研结论

### 1. Vale

来源：`https://github.com/vale-cli/vale`

可借鉴点：

- 把自然语言质量检查做成“规则 + 严重级别 + 提示 + 修复建议”的结构。
- 支持 markup-aware 检查，规则不应误伤标题、代码块、表格、引用等非正文区域。
- 输出结果用 alert 表达，适合后续接入报告、CI 或人工修订清单。

不直接照搬：

- 不引入 Go 运行时或完整 Vale 规则系统。
- 不先追求跨格式通用 lint，先服务 `output-layer` 的 Markdown -> docx 交付场景。

### 2. textlint

来源：`https://github.com/textlint/textlint`

可借鉴点：

- 核心引擎不绑定默认规则，质量标准通过 preset 组合。
- Markdown/plain text 是一等输入，适合 output-layer 当前的源格式。
- 支持 `--fix`、dry-run、formatter 等工作流，后续可参考做“只报告 / 生成修正版 / 渲染成品”三种模式。

不直接照搬：

- 不新增 Node 工具链依赖。
- 不把所有规则都写成低层文本规则；逻辑性、完整性需要文档级检查，不能只靠正则。

### 3. textlint 技术写作预设

来源：`https://github.com/textlint-ja/textlint-rule-preset-ja-technical-writing`

可借鉴点：

- 规则按文体场景组织，而不是只按语法点堆叠。
- 对句长、标点、语体一致、连接词重复、双重否定等设定可调阈值。
- 明确承认“规则会有例外”，需要 allowlist 或人工确认。

不直接照搬：

- 日文规则不能直接迁移到中文正式文档。
- 中文正式文档更关注层级、依据、结论、行动项、责任边界和格式可交付性。

### 4. AI 写作模式检测预设

来源：`https://github.com/textlint-ja/textlint-rule-preset-ai-writing`

可借鉴点：

- 目标不是“限制表达”，而是用结构约束减少 AI 味。
- 对机械化列表、夸张表达、过度强调、英文式冒号续接等模式做检测。
- 规则文档同时给出“检测样式”和“更自然的替代表达”，这对 output-layer 很重要。

不直接照搬：

- 中文正式文件不是“自然口语化”，而是“正式但不空泛”。
- 不应把所有强调和列表都判错；正式文件中表格、条款、编号列表经常是必要结构。

### 5. Vale Microsoft 规则包

来源：`https://github.com/vale-cli/Microsoft`

可借鉴点：

- 规则包可以独立于引擎维护。
- 每条规则应有独立 fixture 或样例，避免规则更新后质量回退。
- 风格指南可以转成机器可读规则，但必须保留人工说明。

不直接照搬：

- 英文技术写作风格不能直接等价于中文正式 docx。
- 该仓库抓到的实现样本较少，只作为规则组织方式参考。

## 我们自己的取舍

推荐路径：先做 `docx-quality-layer-v1`，用“质量门 + 规则预设 + 源库索引”三件套承接。

暂不推荐：

- 直接接入 Vale/textlint。会引入多工具链和跨语言维护成本，而且中文正式文档规则还没沉淀。
- 直接抓取大量正式文档全文。版权、版本、清洗和维护成本都高，第一版只需要源库索引和抽样提炼规则。
- 一上来做 LLM 自动评分。没有规则基线时，评分会不稳定，也不利于复核。

## 第一版质量门

`docx` 输出在渲染前至少经过五类检查：

1. **语体正式性**：不用口语、营销夸张、情绪化、AI 模板句、随意 emoji 或过度强调。
2. **逻辑性**：结论、依据、过程、边界、行动项之间能对应；没有跳结论、互相矛盾或因果缺口。
3. **结构性**：标题层级清楚，段落有职责；列表、表格、引用、附录不混乱。
4. **完整性**：按文档类型检查必要要素，不能只“看起来完整”。
5. **docx 可交付性**：Markdown 标记不会泄漏到 docx；标题、正文、表格、引用、编号、参考文献能被稳定渲染。

## 后续接入建议

第一阶段只做“规则说明 + 人工复核清单 + 可选 YAML 预设”。

第二阶段再把规则分为：

- deterministic：能用脚本稳定检测，例如标题层级、空标题、过长句、模板词、emoji、Markdown 残留。
- assisted：需要模型辅助判断，例如逻辑跳跃、论证不足、完整性缺口。
- human-gate：必须人工确认，例如事实真实性、机构口径、敏感措辞、是否符合特定申请书要求。
