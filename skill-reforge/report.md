# Skill 改造报告

## 1. 改造模式

**推荐模式：B. 工作流贴合版（Workflow-fit）**

**理由：**
1. 原 Skill（yao-meta-skill）的工程方法论骨架（边界设计、触发优先、迭代哲学、质量门控）质量极高，可直接继承。
2. 用户的核心需求与原 Skill 有结构性重叠：都是围绕"如何设计一个好 Skill"。
3. 但场景完全不同——原 Skill 是"从 0 创建新 Skill"，用户需要的是"把已有 Skill 改造成适合自己的"。
4. 原 Skill 的 Python 脚本链（30+ 脚本）、Makefile、CI/CD、多平台打包、治理体系，对个人用户的 Claude Code / OpenClaw 环境来说过重，必须替换为 agent 可直接执行的纯文本流程。

**不选择 A（保守适配）的原因：**
- 原 Skill 的核心流程（创建新 Skill）与用户目标（改造已有 Skill）根本不同，不能简单小幅修改。
- 原 Skill 的脚本链和治理体系对个人用户是噪音而非资产，保留它们会让输出臃肿。

**不选择 C（深度重构）的原因：**
- 原 Skill 的方法论骨架已被验证有效，彻底推翻会增加不必要的风险和开发时间。
- 用户不需要"发明一套全新的 Skill 改造系统"，而是需要"把好的方法论适配到改造场景"。

## 4. 重构内容

| 重构项 | 原形态 | 新形态 |
|---|---|---|
| 核心定位 | "从 0 创建 Skill 的元技能" | "改造已有 Skill 的架构师" |
| 目标用户 | 团队级 Prompt Engineer、内部工具团队 | 个人用户，使用 Claude Code / OpenClaw |
| 输入类型 | 工作流描述、笔记、聊天记录 | 已有 Skill（GitHub 链接、SKILL.md 文本、文件夹） |

| 意图对话 | 5-7 个温暖的问题，强调"老师/教练"语气 | 最多 5 个集中问题，分"必须问"和"不必问"两类，默认自动决策 |

## 6. 边界条件

### 使用边界

- **应该触发**：用户拿着一个别人的 Skill（链接/文本/文件夹）说"帮我改成适合我的"
- **不应该触发**：用户从头描述工作流想变成 Skill；用户只想要 Skill 的总结/评价
- **看起来相关但不该交**：单行格式调整、typo 修复
- **需要用户补充**：仅提供 GitHub 链接但没有给出任何个人偏好时——但此时默认生成可用版本，标注待确认项

### 输入边界

| 输入项 | 状态 | 说明 |
|---|---|---|
| 原 Skill 内容 | 必须 | 缺失时尝试获取，失败则要求用户粘贴 |
| 用户目标需求 | 必须 | 缺失时从上下文推断 |
| 运行环境 | 非必须 | 默认 Claude Code / OpenClaw |
| 目标工具 | 非必须 | 默认 Markdown 输出，无脚本依赖 |
| 输出格式 | 非必须 | 默认 Markdown / Obsidian |
| 文件目录规则 | 非必须 | 默认 `./prompts/` |
| 权限边界 | 非必须 | 默认不执行脚本、不覆盖文件 |
| 是否允许联网 | 非必须 | 仅用于获取原 Skill |
| 是否需要日志 | 非必须 | 改造报告中包含 |
| 是否需要测试 | 非必须 | 自检清单替代 |
| 是否需要示例 | 非必须 | 默认提供 3 个 |

### 工具边界

- 新 Skill 不依赖任何外部命令行工具
- 新 Skill 不依赖 Python / Node / Shell / API
- 网络仅用于获取原始 Skill 内容
- 无危险操作（纯文本生成）
- 适合 Claude Code 和 OpenClaw 执行

### 输出边界

- 必须输出：完整 SKILL.md + 改造报告
- 可选输出：文件树分析（仅当原 Skill 结构复杂时）
- 不输出：代码脚本、HTML 报告、多平台适配文件、治理元数据

### 安全边界

- 写文件前检查路径是否存在
- 不覆盖已有文件（版本化写入）
- 不迁移密钥/路径/账号信息
- 不隐藏不确定性（待确认项机制）
- 不伪造验证结果

## 7. 待确认项

以下假设基于用户提供的偏好和上下文生成，如果与实际情况不符，需要用户确认：

1. **目标目录**：默认输出到 `./prompts/` 目录。如果用户的 Skill 库使用其他目录结构（例如 `~/.claude/skills/` 或项目级 `./skills/`），需要调整。
2. **远程服务器集成**：用户记忆显示有 SSH 访问远程服务器（userysys@100.113.27.115）。如果改造后的 Skill 需要涉及远程文件操作，安全规则需要额外强化。
3. **OpenClaw 具体版本**：假设 OpenClaw 的 Skill 格式与 Claude Code 兼容（单一 Markdown 文件）。如果 OpenClaw 需要特定 frontmatter 或目录结构，需要调整。
4. **多语言偏好**：用户的工作语言似乎是中文为主、英文技术术语混合。SKILL.md 采用中英混合撰写。如果用户偏好纯中文或纯英文，需要调整。
5. **原 Skill 的评估标准**：用户提到"验收标准：分析逻辑自洽；结论有依据；代码可运行；安全审查通过；文档结构完整；术语使用一致；审查意见已处理；用户已确认最终输出"。如果这不是对"改造 Skill"本身的验收标准，而是对某个具体项目的标准，需要澄清。

## 8. 当前风险

| 风险 | 等级 | 说明 | 缓解措施 |
|---|---|---|---|
| 上下文溢出 | 中 | 如果原始 Skill 非常大（如 yao-meta-skill 本身有 30+ 脚本和 20+ 文档），单次对话可能无法完整摄入 | Phase 2 中要求先摘要顶层结构，再请用户选择重点章节 |
| 隐含假设遗漏 | 中 | 原作者的环境假设可能隐藏在某些细节中，改造时可能遗漏 | 强制分类步骤中的 `unknown` 标签 + 待确认项机制 |
| 模式选择错误 | 低 | 自动选择 B（workflow-fit）可能不适用于某些极端情况 | 在 Phase 3 中要求显式宣布模式并给出理由；如果发现冲突，暂停并征求用户确认 |
| 过度简化 | 低 | 删除原 Skill 的脚本链和治理体系后，某些团队级功能丢失 | 明确告知用户这是为个人工作流优化的版本；如果未来需要团队级功能，建议重新评估 |
| 文件覆盖 | 低 | 写入 `./prompts/` 时可能覆盖已有文件 | 安全规则要求写入前检查，存在时版本化 |

## 9. 下一步建议

1. **试用验证**：将这份 `skill-reforge` Skill 实际应用于 1-2 个真实的第三方 Skill 改造任务，验证流程长度和自检机制是否恰到好处（不过度、不缺失）。
2. **触发优化**：根据实际使用中的路由混淆情况，微调 `description` 中的触发关键词，确保它不会被"从头创建 Skill"的请求误触发。
3. **模板化**：如果用户经常改造特定类型的 Skill（例如"把 GitHub 上的 Claude Code Skill 改成 OpenClaw 格式"），可以基于本 Skill 生成一个更具体的子模板，进一步减少每次的重复判断工作。
