# PRD 细节卡

## 什么时候用

当 PRD 只有功能名，但缺页面、步骤、按钮、状态、输入输出或会影响产品判断的数据位置时使用。

## 解决什么缺口

对应 PRD Quality Gate 里的 Detail。

## 操作步骤

1. 把功能改写成用户步骤。
2. 每一步写清用户看到什么。
3. 每一步写清用户能点什么、输入什么、选择什么。
4. 每一步写清系统输出什么。
5. 给每个步骤补状态：空、加载、成功、错误。
6. 把会影响用户理解、验收、权限或风险的数据写成字段和存储位置。
7. 把 API、数据库、目录、组件层级等工程细节移到 ADR / implementation note；只有会改变产品判断时才留在主 PRD。

## 先问用户什么

- 你希望用户从哪里开始？
- 你最在意哪一步体验顺不顺？
- 哪些状态一定不能让用户困惑？

## 不要问用户什么

不要让用户写组件树、API schema、数据库表、队列、错误码或文件目录。用户只需要确认体验、结果、权限、数据风险和不能接受的状态。

## 输出格式

```text
细节补全：
- 用户步骤：
- 页面 / 区域：
- 按钮 / 命令：
- 输入：
- 输出：
- 状态：
- 数据字段：
- 存储位置：
- 主 PRD 保留：
- 附录 / ADR / implementation note：
```

## 质量门

如果“支持某功能”不能被改写成具体步骤、按钮、状态、关键数据位置和验收，说明细节仍不合格。工程级细节不应成为主 PRD 变长的理由。

## 来源 / 借鉴

- Open Design Kit 的非专业用户友好写法。
- React Thinking in React 和 Storybook 的状态拆解思想。
