# PRD 验收路径卡

## 什么时候用

当 PRD 写了目标，但没有说用户怎么验证“做完了”时使用。

## 解决什么缺口

对应 PRD Quality Gate 里的 Acceptance。

## 操作步骤

1. 找出最核心用户场景。
2. 写一条最短路径：打开哪里 -> 做什么 -> 看到什么。
3. 写成功标准。
4. 写失败标准。
5. 写至少一个边界状态：空数据、错误输入、无结果、重复操作。
6. 对核心验收场景补 `Given / When / Then`。
7. 标注哪些验收依赖真实数据、页面实测或用户确认。
8. 写 Stop Conditions：哪些条件不满足时不能说 PRD Ready，不能进入执行。

## 输出格式

```text
验收路径：
- 验收者：
- 起点：
- 操作步骤：
- 成功结果：
- 失败 / 错误状态：
- 边界用例：
- Given / When / Then：
- 依赖数据：
- 通过标准：
- Stop Conditions：
```

## 不要做

- 不要只写“功能正常可用”。
- 不要用技术测试替代用户验收。
- 不要把所有测试都塞进 PRD，PRD 只保留最短验收路径和关键标准。
- 不要在缺少用户确认、账号权限、真实数据或关键证据时标记 Ready。

## 质量门

一个人照着验收路径操作，应该能在 3-10 分钟内判断第一版是否可用；如果不能判断，PRD 只能标 `Needs clarification`。

## 来源 / 借鉴

- PRD release criteria。
- QA smoke test 和用户任务路径。
