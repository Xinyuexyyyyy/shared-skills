# PRD 可执行规格卡

## 什么时候用

当 PRD 看起来完整，但实现者读完仍不知道第一步怎么开工时使用。

## 解决什么缺口

对应 PRD Quality Gate 的总门：Boundary x Detail x Evidence x Acceptance x Stop Conditions。

PRD 不是执行交接模块，也不是想法说明书。它是产品判定文档：读者应该能判断第一版做什么、不做什么、做到什么样算成功、什么情况下必须停下来问人。

## 操作步骤

1. 用一句话写清第一版用户能看到什么结果。
2. 列出目标用户、核心场景、痛点和不做会怎样。
3. 把 MVP / Future / Out of Scope 分开，MVP 必须写成用户可见结果，不写功能名。
4. 把用户流程写成步骤，不只写能力名。
5. 给每个关键步骤补页面、按钮、状态、输入、输出。
6. 对核心能力补 `Requirement / Scenario`；关键场景用 `Given / When / Then` 写成可验证行为。
7. 给每个会影响产品判断的关键数据补字段和存储位置；文件路径、目录、API、数据库等工程细节默认进入 ADR / implementation note。
8. 写最短验收路径：从哪里开始、做什么、看到什么算成功。
9. 把风险写成 Stop Conditions：什么情况下不能继续写 PRD、不能进入执行、必须问用户。
10. 标注每个关键假设的证据或验证动作。

## 输出格式

```text
可执行规格摘要：
- 用户：
- 场景：
- MVP：
- 不做：
- 关键流程：
- Requirement / Scenario：
- UI 行为：
- 数据与存放位置：
- 附录 / ADR / implementation note：
- 验收路径：
- Stop Conditions：
- 证据 / 假设：
```

## 不要做

- 不要把调研过程塞进 PRD 正文。
- 不要用“支持 X”“优化 Y”代替具体行为。
- 不要把待定项伪装成已确定。
- 不要把 API、数据库、目录、组件层级等工程细节塞进主 PRD，除非它们会改变范围、验收、权限、数据风险或 Stop Conditions。

## 质量门

一个实现者只看 PRD，应能回答：

- 第一版做什么、不做什么？
- MVP 是用户可见结果，还是功能名？
- 页面上有什么按钮和状态？
- 核心行为能否用 Given / When / Then 验证？
- 用户数据存在哪里、是否有权限/隐私/丢失风险？
- 工程细节是否已放到附录 / ADR / implementation note，而不是压进主 PRD？
- 我怎么验证它做完了？
- 什么情况下必须停下来问人？

## 来源 / 借鉴

- PRD/SRS 模板中的可验证需求字段。
- Atlassian 产品需求中的目标、假设、用户故事、验收思想。
