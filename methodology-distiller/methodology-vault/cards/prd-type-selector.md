# PRD 类型选择卡

## 什么时候用

当用户说“写 PRD”“做方案”“整理需求”，但输入还不确定是否足以支撑完整 PRD 时使用。

## 解决什么缺口

防止把所有内容都硬包装成 PRD。对应 PRD Quality Gate 里的 Boundary、Evidence、Acceptance。

## 判断规则

| 情况 | 交付物 |
|---|---|
| 只有模糊想法，用户/痛点/成功标准不清 | `idea_note` |
| 方向存在，但证据和关键假设不足 | `discovery_note` |
| 主要是在比较方案或做取舍 | `decision_brief` |
| 用户只是要理解材料或网页 | `reader_brief` |
| 用户主要需要来源、竞品表、价格表、页面核验 | `evidence_brief` |
| 用户、边界、细节、证据、验收都足够 | `PRD` |

## 先问用户什么

- 这个产物给谁看？
- 写完后希望别人做什么？
- 现在最怕写错的是方向、范围、实现，还是验收？

## 不要问用户什么

不要一开始就问“数据库怎么设计”“接口怎么设计”“组件怎么拆”。这些属于后续方法蒸馏。

## 输出格式

```text
交付物判断：
- 推荐类型：
- 不推荐直接写 PRD 的原因：
- 如果要升级成 PRD，需要补：
- 当前可以交付：
```

## 质量门

读完后应该能判断：现在该写完整 PRD，还是先降级成 note / brief。

## 来源 / 借鉴

- PRD / Feature Brief / Epic 类型区分来自公开 PM toolkit 和产品管理实践。
- 降级出口接入本地 `deep-research` Phase 5。
