# PRD Grok 追问卡

## 什么时候用

当 PRD Quality Gate 已经发现缺口，但直接批注“这里不清楚”无法推动补齐时使用。

典型场景：

- Boundary、Detail、Evidence、Acceptance 有缺口。
- 用户给的是理念、愿景、粗功能名或混乱草稿。
- 需要把“问题发现”回滚成一组递进追问。
- 不能直接重写 PRD，也不能编造用户没确认的业务事实。

## 解决什么缺口

只把 `Owner = User` 的 PRD Gate 失败项变成可回答、可回填、可验证的问题链。

它不负责写完整 PRD，只负责把缺口压成 3-7 个高质量追问，并说明每个答案要补回 PRD 的哪个字段。

## 总原则

- 一次只追一个最致命缺口，不撒问卷。
- 每个问题都必须对应一个 PRD 字段。
- 问用户业务目标、核心取舍、真实场景、体验预期。
- 不问用户架构、数据库、API schema、组件拆分。
- `Evidence / Method / Spec` 缺口不进入追问链：分别转 `deep-research`、`methodology-distiller`、PRD detail/acceptance/subtraction pass。
- 用户答不上来的专业方法，转 `methodology-distiller` 或默认假设。
- 每一轮追问后，都要说明“这个答案会补回哪里”。

## 缺口到追问映射

| Gate 缺口 | 追问方向 | 补回字段 |
|---|---|---|
| Boundary 不清 | 第一版必须解决哪个场景？哪些明确不做？未来想象哪些先放下？ | MVP / Out of Scope / Future / Non-goals |
| Detail 不清 | 用户在哪个页面？点哪个按钮？输入什么？看到什么？失败时看到什么？ | 用户步骤 / 页面 / 按钮 / 输入输出 / 状态 |
| Evidence 不清 | 这个判断来自哪里？有没有真实材料？没有的话是先标假设还是补证据？ | 证据 / 假设 / 待验证 |
| Acceptance 不清 | 从哪里打开？做哪一步？看到什么算成功？什么算失败？ | 验收路径 / 成功标准 / 失败标准 / 边界用例 |
| Priority 不清 | 如果只能先做一个能力，哪个不做会让第一版失去价值？ | 优先级 / MVP wedge |
| Taste 不清 | 用户看到什么会觉得“能用”？什么会觉得“虽然实现了但不好用”？ | 体验标准 / 状态文案 / 交互取舍 |

## 操作步骤

1. 读取 PRD Gate 失败项，找出最影响开工的一个缺口。
2. 把缺口改写成一句“为什么这会阻塞 PRD”的话。
3. 生成 3-7 个递进追问。
4. 每个追问后标注“回答后补回 PRD 的字段”。
5. 标出哪些问题必须用户回答，哪些应由系统蒸馏或默认。
6. 如果用户回答后仍不够，继续追问同一条链，不跳到新话题。
7. 当答案能回填字段时，停止追问，回到对应方法卡或 PRD Quality Gate。

## 输出格式

```text
Grok 追问回滚：
- Gate 缺口：
- Owner：User
- 为什么阻塞：
- 追问链：
  1. 问题：
     补回字段：
     谁来回答：用户 / 系统蒸馏 / 默认假设
- 不进入追问的缺口：
  - Evidence：
  - Method：
  - Spec：
- 当前不能编造：
- 回填后回到：
```

## 好问题标准

好问题应该满足：

- 用户能用自己的业务语言回答。
- 答案能直接写进 PRD。
- 不要求用户懂技术实现。
- 能把模糊词压成可观察行为。
- 能帮助判断“做 / 不做 / 以后做 / 先假设”。

## 不要做

- 不要把 PRD 批注写成“建议补充 XX”就结束。
- 不要一次列 20 个问题。
- 不要问“数据库怎么选”“后端怎么设计”“组件怎么拆”。
- 不要替用户编造业务优先级。
- 不要把所有缺口同时追问，先抓最影响开工的一条。

## 质量门

追问链通过的标准：

- 每个问题都能落到一个 PRD 字段。
- 至少一个问题能直接补齐当前 Gate 缺口。
- 用户无需懂技术也能回答。
- 回答后能回到 `prd-boundary`、`prd-detail`、`prd-data-storage`、`prd-acceptance-path`、`prd-subtraction` 或 PRD Quality Gate。

## 来源 / 借鉴

- 本地 `grillme` 的一问一答和强问题规则。
- PRD Quality Gate 的 Boundary / Detail / Evidence / Acceptance 失败项。
- 产品访谈中的追问法：先追真实场景，再追取舍，再追可验收结果。
