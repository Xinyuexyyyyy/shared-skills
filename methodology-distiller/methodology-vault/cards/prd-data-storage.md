# PRD 数据与存储卡

## 什么时候用

当 PRD 没写清数据字段、数据放哪里、数据如何更新、谁有权限、是否可丢失或是否需要备份时使用。

## 解决什么缺口

补 PRD Detail 中会影响产品判断的数据决策；工程级 schema、目录和迁移细节默认进入 ADR / implementation note。

## 先问用户什么

- 哪些数据你不能丢？
- 哪些数据只是临时过程，可以重新生成？
- 你更在意简单可用，还是长期可迁移？
- 你需要手动编辑这些数据吗？

## 系统自己要蒸馏什么

- 实体、字段、关系、生命周期。
- 本地文件、数据库、JSON/Markdown、表格等存储方式的取舍。
- 目录布局和命名约定。
- 最小备份和迁移策略。

## 输出格式

```text
数据与存储：
- 核心实体：
- 字段：
- 存储位置：
- 读写方式：
- 生命周期：
- 备份 / 迁移：
- 主 PRD 保留：
- ADR / implementation note：
- 待验证风险：
```

## 不要做

- 不要让轻代码用户选择数据库范式。
- 不要过早设计复杂权限、同步、迁移系统。
- 不要只写“保存到数据库”，必须写清楚保存什么和在哪里。
- 不要把 schema、目录和迁移方案塞进主 PRD，除非它们会改变范围、验收、权限、数据风险或 Stop Conditions。

## 质量门

用户应该知道自己的数据在哪里、会不会丢、谁能访问、出错时怎么办。实现者需要的文件/表/字段可以放在 ADR / implementation note。

## 来源 / 借鉴

- Prisma / MongoDB 数据建模中的实体、访问模式、生命周期思想。
- PostgreSQL 约束中的数据完整性思想。
