# Workspace Tidy — 工作区整理优化

## 定位

工作区跑一段时间后，根目录会堆裸文件、临时目录、很久没动的旧项目。
这个命令扫描当前一级目录，**给出整理方案，等用户确认后才动手**，把工作区收归到约定结构。

**目录约定 + 归类规则的唯一事实源**：`skills/workspace-layout.md`。

## Outcome Contract

- **产出**：一张「文件/目录 → 建议动作」的整理方案表 + 用户确认后的执行结果
- **完成标准**：用户确认的项被安全移动到位，未确认的一律不动
- **证据**：每一步实际的 mv 结果 + 一份 move log（可回滚）

## 红线（最重要，必须遵守）

- **绝不自动删除**任何文件。要删的只"标记待删"，列出来让用户自己删
- **移动前必须出方案、等用户明确确认**，绝不先斩后奏
- **归档 = 移到 `archive/`**，不是删除
- 拿不准的一律标"待定"问用户，不擅自归类
- 每次移动都记进 move log，方便回滚

## 工作流

### Step 1：扫描当前一级目录
- `pwd` 确认位置；`ls -la` 列一级目录
- 若约定目录（research/ output/ archive/ 等）不存在，提示"建议先跑 `/workspace-init`"

### Step 2：按规则分类（读 workspace-layout.md 的「归类规则」）
把一级项归成几类：
- **垃圾**：`.DS_Store`/`*.tmp`/空目录 → 标记待删（不删）
- **裸文件**：根目录单文件 → 按扩展名建议归到 docs/data/resources/scripts
- **该归档**：临时目录(30天未动) / 旧项目目录(90天未动) → 建议移到 archive/
- **待定**：拿不准的，列出来问
- **不动**：已经在约定目录里的、正在用的

判断"多久没动"用 mtime：
```bash
find . -maxdepth 1 -mtime +90 -type d   # 90天没动的目录
```

### Step 3：出整理方案（停下，等确认）
打印一张方案表，**然后停下，不要执行**：
```
项目                  类型      建议动作                   理由
navigation.app.py     裸文件    → scripts/                 根目录脚本
research_tmp/          临时目录  → archive/                 30天未动
.DS_Store              垃圾      标记待删（需你手动删）     系统垃圾
某旧项目/              旧目录    → archive/                 90天未动
某文件                 待定      ？问你                     拿不准
```
明确告诉用户：「以上是建议，确认哪些就动哪些。回复『全部确认』或挑选要执行的项；待删项我不会删，只列出来。」

### Step 4：执行（只动确认的）
- 用户确认后，逐项 `mv`（移动优先）
- 目标目录不存在先 `mkdir -p`
- **同名冲突**：目标已有同名文件 → 不覆盖，加后缀或问用户，绝不直接盖
- 标记待删的项：**只打印路径，不执行删除**，让用户自己删

### Step 5：记 move log + 报告
- 把每条 `mv 源 目标` 写进 `archive/.tidy-move-log-<日期>.txt`（用户提供日期或跳过时间戳），方便回滚
- 报告：移了几项、跳过几项、待删几项（待用户处理）
- 给一句回滚提示：照 move log 反向 mv 即可还原

## 它在防什么坑

- 工作区跑久了根目录全是裸文件和旧东西，找不着北
- AI 擅自移动/删除用户文件，移错了找不回
- 黑盒瞎归类，把不该归的归错

## 什么时候用

- 工作区根目录变乱，想整理回约定结构
- 一堆临时文件/旧项目堆积，想归档

## 不什么时候用

- 全新空工作区 → 用 `/workspace-init` 先搭骨架
- 想收尾任务 / 更新记忆 → 用 `closeout`

## 与其他 skill 的边界

- `workspace-init`：搭骨架（只新建）；tidy 整理已有的乱（会移动，需确认）
- `closeout`：管任务收尾 + 记忆建议，不碰文件目录

## 实战改进（来自真实试用，必须遵守）

1. **别只信目录 mtime**：目录的 mtime 只反映直接子项变化，深层文件改了它不一定更新。
   判断"多久没动"要看最深层文件：
   ```bash
   find "$d" -type f -exec stat -f '%m' {} \; | sort -rn | head -1   # 目录内最新文件的时间
   ```
   拿不准就归"待定"，不靠 mtime 一刀切。

2. **记住归类决定**：用户对每个目录的归类口述（如"easyai 没用了→archive"）要记进
   `archive/.tidy-decisions.md`，下次 tidy 先读它，已决定过的不再重复问，只问新出现的项。

3. **关联文件一起动**：同名不同扩展的文件（如 `navigation.app.py` / `navigation.index.html`）
   多半是一套，方案里要把它们**放在一起建议**，并提示用户"这是一对，建议同去一处"，
   不要默认拆散到两个目录。
