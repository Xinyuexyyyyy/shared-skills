# Task Analyze — 意图分析

## 定位

任何新需求、模糊想法、问题描述的第一站。

它不负责直接执行，而是先把“用户到底想干什么”说清楚，再决定是不是该进入下一步。

它的目标不是把事情规划得很大，而是先完成一个轻量判断：

- 这件事真正要达成什么效果
- 当前明确不做什么
- 关键约束和风险在哪里
- 是否值得继续拆解或执行
- 做完后用户能看到什么、怎么验收

## 它在防什么坑

- 需求还没说清，就直接开做
- 用户问的是 A，结果做成了 B
- 一个输入里其实有多个目标，但没有先分优先级
- 应该走专属 skill 的任务，被当成普通需求硬分析
- 一上来就扩成项目、系统、长期规划
- 只讲技术方案，不讲用户可见效果
- 没确认风险点就进入执行或文件操作

## 什么时候用

- 用户提出新想法、描述问题、给出需求
- 用户分享信息、询问方案
- 用户说“分析一下”“帮我看看”“想做个东西”

## 不什么时候用

- 用户明确要求直接执行，而且没有明显歧义
- 用户只是闲聊、确认或简单回应
- 用户的问题已经明显属于某个专属 skill
- 当前任务已经分析清楚，正在执行或收尾

## 默认工作方式

先判断，不急着给方案，也不急着拆任务。

如果存在关键分歧或多个目标，就先问清优先级；  
如果目标已经足够清楚，就直接给出结构化意图结果，并建议进入下一步。

默认采用“目标 -> 边界 -> 取舍 -> 验收 -> 下一步”的轻量判断链：

1. **目标**：用户真正想达成的可见效果。
2. **非目标**：本轮明确不做什么，防止范围外扩。
3. **约束**：时间、路径、工具、权限、输出格式、质量要求。
4. **取舍**：如果存在多条路线，指出推荐方向和放弃什么。
5. **验收**：做完后用户如何确认这件事完成。
6. **下一步**：进入执行、拆解、专属 skill、补信息，或不做。

除非用户明确要完整方案，否则不要输出长方案；只输出足以决定下一步的信息。

## 边界条件

**条件**：用户明确要求直接执行（"写段代码"、"跑一下测试"）  
**处理**：跳过分析，直接执行，不再输出意图报告

**条件**：用户仅做闲聊或确认（"好的"、"明白了"）  
**处理**：不触发本 skill，正常对话即可

**条件**：输入为空或完全无法理解  
**处理**：回复"请补充更多信息，我无法判断您的意图"

**条件**：需求明显属于已有 skill 的专属领域（如 interview-writer）  
**处理**：转交对应 skill，不再重复分析

**条件**：涉及多个目标且用户未说明优先级  
**处理**：列出主要目标，询问"您希望优先处理哪一个？"

**条件**：用户提出的是新想法、产品方向或“值不值得”  
**处理**：先做轻量价值判断，输出推荐方向、放弃方向和需要什么证据改变判断；不要直接进入实现拆解

**条件**：任务涉及删除、覆盖、移动、部署、发布、安装、全局配置、账号、密钥、隐私或商业敏感数据  
**处理**：标为高风险确认点，不进入执行；说明需要用户明确授权的动作

**条件**：用户想做的事超过一个会话或超过 5 个子任务  
**处理**：先建议收敛到最小闭环，再交给 `task-decompose` 或 `task-crafter`

**条件**：用户想要的是资料读取、UI 设计、写作输出、学习闭环等独立专业面  
**处理**：推荐对应专属 skill 或候选 skill，不把它们塞进普通 task 分析

## 工作流（4 步）

### Step 1：提取核心意图

用一句话回答“用户真正想做什么”。

- 如果输入为空或仅有语气词 → 终止，请求补充
- 如果涉及多个目标 → 列出主要目标，询问优先级
- 如果用户的字面请求和真实目标可能不同 → 先写真实目标，再说明推断依据

### Step 2：分类与约束扫描

| 维度 | 输出 |
|------|------|
| 任务类型 | 调研 / 实现 / 决策 / 排查 / 整理 / 其他 |
| 已知约束 | 时间、技术栈、预算、范围 |
| 缺失信息 | 推进前必须确认的事项 |
| 非目标 | 本轮明确不做的内容 |
| 风险点 | 可能导致误做、越权、返工或范围扩张的事项 |
| 验收方式 | 用户做完后能看到什么、怎么确认 |

类型判断标准：
- 调研："看看""了解""调研""对比"
- 实现："做""写""搭""开发"
- 决策："选哪个""要不要""怎么选"
- 排查："为什么""报错""修"
- 整理："总结""梳理""整理"
- 设计："UI""页面""组件""不好看""不协调"
- 读取："链接""PDF""网页""读一下"
- 学习："学习""掌握""章节""课程""陌生领域"
- 写作："写文案""改稿""润色""去 AI 味"

### Step 3：做取舍判断

只在必要时输出。满足任一条件时必须做：

- 用户问“要不要、值不值得、怎么选”。
- 存在多个互斥路线。
- 继续做会引入新工具、长期维护、外部服务或复杂架构。
- 用户的想法容易变成大项目。

输出：

- **推荐方向**：当前最小可行方向。
- **不推荐方向**：为什么先不做。
- **判断依据**：来自用户目标、现有工作区、风险或成本。
- **改变判断的证据**：什么新信息会让推荐变化。

### Step 4：推荐下一步

- 信息充足 → "建议进入 task-decompose 拆解"
- 信息缺失 → "建议先确认：[列表]"
- 范围太大 → "建议收敛到最小范围"
- 复杂度超过 1 天 → "建议走 task-crafter 完整规划"
- 明显属于专属 skill → "建议进入 [skill-name]，原因是..."
- 风险动作未授权 → "需要先确认权限：[动作]"
- 不值得做 → "建议不做 / 暂缓，并说明替代动作"

## 用户可见检查点

### 检查点 1：先说核心目标

先用一句话说清：
- 我理解你真正想做什么

### 检查点 2：把缺失信息挑出来

如果当前不能直接推进，要明确说：
- 还缺什么
- 不确认会导致什么问题

### 检查点 3：说清边界和验收

必须能回答：
- 本轮不做什么
- 做完后用户能看到什么
- 用户怎么确认完成

### 检查点 4：下一步必须具体

最后不能只说“我知道了”，而要明确给出：
- 进 `task-decompose`
- 进专属 skill
- 还是先补信息
- 还是建议暂缓 / 不做

## 输出格式

```markdown
## 意图分析
**核心目标：** [一句话]
**任务类型：** [类型]
**置信度：** [高/中/低]

## 本轮边界
- **只做：** [本轮聚焦]
- **不做：** [明确排除]

## 已知约束
- [约束1]
- [约束2]

## 缺失信息
- [缺失1]（若不确认可能导致...）

## 取舍判断
- **推荐：** [推荐方向]
- **暂不推荐：** [不推荐方向]
- **依据：** [一句话]

## 验收方式
- [用户可见结果或检查方式]

## 推荐下一步
[建议，并说明原因]
```

如果任务很小，可以压缩输出，但不得省略核心目标、边界、验收和下一步。

## Skill 关联

| 情况 | 下一步 Skill |
|------|-------------|
| 信息充足 + 可当天完成 | skill: task-decompose |
| 信息充足 + 跨天/跨模块 | skill: task-crafter |
| UI / 页面 / 组件 / 视觉问题 | skill: ui-design（若已安装）或进入 UI 设计候选流程 |
| URL / PDF / 网页 / 资料读取 | skill: source-intake（若已安装）或 research 前置读取流程 |
| 陌生领域 / 章节学习 / 学习闭环 | skill: study-loop（若已安装）或笔记驱动学习引擎 |
| 写作 / 改稿 / 文案 / 去 AI 味 | skill: output / interview-writer / write-polish 候选流程 |
| 报错 / 异常 / 回归 | 排查模式：先确认现象和根因，再决定是否执行 |
| 任务已完成 | skill: closeout |

## Action

- `analyze` — 主流程，分析用户输入并输出意图报告

## 最小验收

这轮至少要做到：

1. 用一句话说清核心目标
2. 给出任务类型和已知约束
3. 说明本轮只做什么、不做什么
4. 只列最关键的缺失信息和风险点
5. 说明做完后用户怎么验收
6. 下一步要落到具体 skill、具体确认动作、暂缓或不做
7. 不替用户拍板，不顺手扩展需求
