# UI Design — 界面设计与截图验收

## 定位

把页面、组件、工具界面或截图反馈改成用户能看、能用、能验收的 UI。

它不只是“改样式”，而是先判断界面要服务谁、用户要完成什么动作，再围绕布局、层级、密度、状态、响应式和真实截图做改造。

## Outcome Contract

- **产出：** UI 设计判断、具体改造方案、实现后的界面检查，或可交给执行 agent 的设计任务说明。
- **完成标准：** 用户能看到真实页面/截图/组件变化，并知道怎么确认它比原来更清楚、更好用。
- **证据：** 当前截图、页面 URL、本地组件文件、设计约束、用户反馈、浏览器检查结果或明确的未验证说明。
- **输出：** 先讲用户可见效果，再讲技术实现；不输出空泛审美建议。

## 它在防什么坑

- 页面能跑，但看起来像临时拼出来的
- 只改颜色和圆角，没有解决信息层级
- 桌面好看，移动端溢出或重叠
- 卡片套卡片、装饰过度、营销页风格误用到工具界面
- 按默认模板生成，没有产品气质
- 没看真实截图就声称 UI 已完成

## 什么时候用

- 用户说 UI、页面、组件、前端、界面、视觉、交互
- 用户给截图说“不好看”“很怪”“不协调”“不清晰”“突兀”
- 需要做 dashboard、工具台、编辑器、表单、列表、设置页等工作型界面
- 需要检查页面在桌面/移动端是否可用
- 需要把一个已有页面做视觉和交互 polish

## 不什么时候用

- 纯后端、数据管道、脚本任务
- 只问产品方向，还没到界面层，应先走 `task-analyze`
- 只做文案润色，应走 `output` / `write-polish`
- 只生成文章配图、封面图、插画，不涉及 UI
- 完整品牌系统、logo、VI 设计，除非用户明确要求
- 没有任何界面目标或用户流程，只想“随便变好看”

## 默认工作方式

先判断界面的使用场景，再做视觉决策。

默认顺序：

1. **用户任务**：这个界面让用户完成什么动作。
2. **界面类型**：工作台、表单、编辑器、内容页、营销页、数据页、设置页。
3. **当前问题**：布局、层级、密度、对齐、状态、文案、响应式、视觉风格。
4. **结构影响**：判断改动是否会改变首屏重心、左右权重、功能存在感或信息密度。
5. **改造范围**：本轮只改什么，不改什么。
6. **实现或交付**：按项目现有技术栈和组件风格改。
7. **截图验收**：桌面和移动端至少各检查一次；不能检查就说明未验证。

## 设计原则

### 1. 工作型界面优先可扫读

SaaS、CRM、后台、工具台、学习工具、运营系统应保持安静、密集、可重复使用。

避免：

- 过大的 hero
- 装饰性卡片堆叠
- 大面积单色渐变
- 营销页式空洞标语
- 不必要的插画和氛围背景

### 2. 页面先有信息层级

每个界面至少明确：

- 主操作
- 次操作
- 当前状态
- 关键数据
- 错误/空/加载状态
- 用户下一步

如果层级说不清，不要先改颜色。

### 3. 组件要有稳定尺寸

固定格式元素必须定义稳定尺寸或约束：

- 工具栏
- 图标按钮
- 棋盘/网格/卡片
- 列表行
- 输入框
- 计数器
- 状态标签

避免 hover、loading、长文本、图标切换造成布局跳动。

### 4. 文本不能溢出或遮挡

必须检查：

- 按钮文字是否放得下
- 长词是否撑破容器
- 中英文混排是否挤压
- 移动端是否换行合理
- 文本是否覆盖前后内容

不要用随 viewport 缩放的字体解决问题。

### 5. 视觉风格要贴合产品

根据界面性质选择风格：

| 类型 | 风格方向 |
|---|---|
| 工具台 / SaaS | 克制、清晰、可扫读 |
| 内容阅读 | 舒适排版、留白、节奏 |
| 创作工具 | 操作高效、状态明确、快捷入口 |
| 数据看板 | 比较、过滤、趋势优先 |
| 游戏 / 玩具 | 可更强表达和动画 |
| Landing | 首屏必须展示真实产品/对象/场景 |

## 工作流

### Step 1：确认界面目标

回答：

- 谁在用
- 在什么场景用
- 要完成什么动作
- 当前最影响使用的 UI 问题是什么

如果目标不清，先问 1 个关键问题，不要问散问题。

### Step 2：读现有界面和约束

优先读取：

- 用户截图
- 页面 URL 或本地运行地址
- 相关组件文件
- 现有 CSS / design token / UI library
- 项目里的相邻页面

不要在没看现有界面的情况下大改风格。

### Step 3：定义本轮边界

输出或心中明确：

- 本轮只改什么
- 本轮不改什么
- 哪些属于后续设计系统问题
- 是否需要用户确认

先判断这轮改动属于哪一类：

| 类型 | 可以直接做 | 必须先确认 |
|---|---|---|
| 局部修正 | 错位、溢出、遮挡、按钮高度、间距、hover/focus 状态 | 无 |
| 局部强化 | 在现有结构内强化主按钮、输入框、提示、空/错/加载态 | 无 |
| 结构性改动 | 无 | 改变首屏左右权重、删减/弱化功能区、改变主操作、重排信息架构、明显增减信息密度 |
| 视觉方向改动 | 无 | 更换整体风格、品牌感、图片资产、字体、色彩主调、UI 框架 |

如果属于结构性改动，先输出：

```markdown
## UI 问题诊断
- 当前问题：[具体问题]
- 影响：[用户会怎么感受到/怎么用错]

## 可选方案
- A. [保守修正]：保留原信息权重，只修 [...]
- B. [结构调整]：改变 [...]，代价是 [...]

请确认选 A 还是 B；确认前我不改结构。
```

用户确认前，只能做不改变结构权重的局部修正。

涉及新增依赖、换 UI 框架、重做布局体系、引入图片资产，也必须先确认。

### Step 4：设计或实现

按任务要求选择：

- 用户要方案：输出可执行设计方案。
- 用户要改页面：直接修改代码。
- 用户给截图说丑：先指出具体视觉问题，再改。
- 用户要审查：输出问题清单和优先级。

改代码时遵守项目现有组件、样式系统和技术栈。

### Step 5：验收

至少检查：

- 桌面布局
- 移动端布局
- 主要交互状态
- loading / empty / error / disabled 状态（如相关）
- 文本是否溢出
- 控件是否可点击、可扫读

如果项目可运行，优先用浏览器或截图验证。不能验证时明确写“未实机验证”。

## 用户确认规则

必须先问：

- 要重做整体信息架构
- 要换 UI 框架或设计系统
- 要新增大图、字体、图标库、三方组件
- 要删除现有功能入口
- 要弱化、收起、缩小或移走用户已经认可的功能区
- 要改变首屏左右权重、主次关系、信息密度或主要操作路径
- 要覆盖用户已有设计稿
- 要引入品牌、logo、版权素材

不必问：

- 修正明显错位、溢出、遮挡、间距问题
- 在现有风格和原信息权重内微调层级、对齐、状态
- 增加必要的 hover/focus/disabled/loading 状态
- 为移动端增加合理响应式约束

## 输出格式

### 设计方案

```markdown
## UI 设计判断
- **用户任务：** [...]
- **当前问题：** [...]
- **本轮只做：** [...]
- **本轮不做：** [...]

## 改造方案
1. [改动]，用户可见效果：[...]
2. [改动]，用户可见效果：[...]

## 验收方式
- 桌面：[...]
- 移动端：[...]
- 状态：[...]
```

### 实现完成

```markdown
## 完成结果
- [用户可见变化]

## 验证
- 桌面：[通过 / 未验证 + 原因]
- 移动端：[通过 / 未验证 + 原因]
- 交互状态：[通过 / 未验证 + 原因]

## 文件
- [路径]
```

### 截图审查

```markdown
## 截图问题
1. [问题]，影响：[用户感受/使用问题]
2. [问题]，影响：[用户感受/使用问题]

## 优先修
1. [最高杠杆改动]
2. [次要改动]
```

## 质量检查清单

- [ ] 用户任务说清楚
- [ ] 本轮边界说清楚
- [ ] 已判断是否改变首屏重心、左右权重、功能存在感或信息密度
- [ ] 如涉及结构性改动，已先给诊断和 A/B 方案并等待确认
- [ ] 主操作和次操作清楚
- [ ] 信息层级清楚
- [ ] 文本不溢出、不遮挡
- [ ] 桌面和移动端都有验收口径
- [ ] loading / empty / error / disabled 状态不破坏布局
- [ ] 没有卡片套卡片
- [ ] 没有无意义装饰背景
- [ ] 没有用技术完成代替用户可用

## 失败处理

| 情况 | 处理 |
|---|---|
| 没有截图或运行环境 | 读代码和相邻页面，标注未实机验证 |
| 页面跑不起来 | 先报告阻塞和已尝试命令，不声称视觉已验收 |
| 用户只说“不好看” | 先指出 2-4 个具体视觉问题，再改 |
| 移动端布局崩 | 优先修响应式约束，不继续加视觉装饰 |
| 设计方向涉及品牌 | 先确认品牌边界 |
| 改动会影响功能入口 | 先确认，不擅自删除 |
| 改动会弱化用户认可的区域 | 先说明问题和可选方案，确认前只做局部修正 |

## 最小验收

这轮至少要做到：

1. 说清界面服务的用户任务
2. 说清本轮只改什么、不改什么
3. 给出用户可见变化
4. 检查桌面和移动端，或明确未验证原因
5. 确认没有未经同意改变用户认可的信息权重
6. 不把“能跑”当成“好用”
