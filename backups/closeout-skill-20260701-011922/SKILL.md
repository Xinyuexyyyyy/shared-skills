# Closeout

## 定位

这是任务结束时的统一收尾入口。

它不负责继续扩任务，只负责把这轮做了什么、解决了什么、还能做什么收干净，并在需要时给出记忆更新建议。

它的核心不是“礼貌总结”，而是交付检查：用户现在能看什么、试什么、怎么确认完成、还有什么没做。

## Outcome Contract

- **产出：** 6 段收尾总结，必要时附清退建议、锚点扫描和记忆写入建议。
- **完成标准：** 用户能一眼看懂本轮结果、验证路径、剩余风险、下一步需要拍什么板。
- **证据：** 真实文件路径、页面地址、命令结果、测试结果、截图结果、远端状态或明确说明“未验证”。
- **输出：** 面向用户的交付状态，不写成内部工作日志，不借收尾扩需求。

## 它在防什么坑

- 事情做完了，但没有留下清楚总结
- 用户不知道现在到底能验证什么
- 当前焦点和待办越来越脏，没人清理
- 没得到确认就擅自写记忆
- 技术完成了，但用户不知道产品是否可用
- 测试、页面、文件、远端状态没有说清，导致结果无法验收
- 收尾时顺手开新坑，破坏 MVP 收敛

## 什么时候用

- 任务完成
- 用户说"先这样""停""总结"
- 会话即将结束
- 任务中途叫停
- 遇到阻塞，需要把已完成和卡点收清
- 完成了文件、代码、部署、调研、文档或 skill 改造等可交付产物

## 不什么时候用

- 任务还在推进中
- 当前只是中间状态，还没到一个自然收口点
- 用户明确要求继续，不需要总结
- 还没有任何可说明的结果、阻塞或用户决策点

## 默认工作方式

先收尾，再建议写记忆。

默认顺序是：
1. 先做交付检查：结果、证据、验证方式、剩余风险
2. 再给 6 段总结
3. 如有需要，做锚点扫描
4. 如有需要，再给清退建议
5. 最后明确询问是否写记忆

收尾前先判断本轮状态：

| 状态 | 处理 |
|---|---|
| 正常完成 | 说明产物、路径、验证结果、下一步 |
| 部分完成 | 说明完成部分、未完成部分、为什么停 |
| 阻塞 | 说明卡点、已尝试动作、需要谁做什么 |
| 用户叫停 | 说明已做、未做、恢复入口 |
| 验证失败 | 说明失败证据、影响、回退或下一步 |

如果没有实际验证，必须写明“未验证”，不能用“应该可以”替代。

## 6 段输出

```markdown
## 收尾总结

### 我要做的事
[本轮做什么，1-2 句]

### 解决的问题
[解决了什么痛点]

### 目标效果
[用户能看到什么变化]

### 现在用户能做什么
[验证路径：文件、页面、命令、截图、远端状态或手动检查方式]

### 未完成内容
[剩余任务]

### 下一步决策
[等人拍板]
```

每段都要讲用户能理解的效果，再补技术细节。不要用“接口完成”“模块完成”代替“用户现在能做什么”。

## 研究模式（Research Mode）

当本轮对话涉及论文、调研、实验、选题、方法设计、文献分析等研究任务时，自动激活研究模式。

研究模式不替代 6 段收尾，只在 6 段总结之后追加“研究探索记录”。不要引入 ARA 完整 schema，也不要写入 `ara/` 目录。

从对话历史提取 7 类事件：

| type | 提取规则 |
|---|---|
| `decision` | 用户在多个选项中选了哪个，或明确否掉哪个方向 |
| `experiment` | 跑了什么实验/验证/测试，结果如何 |
| `dead_end` | 什么路走不通，为什么放弃 |
| `pivot` | 重大方向变更，以及触发变更的证据 |
| `claim` | 关于系统、方法、数据或研究结论的关键断言 |
| `heuristic` | 本轮发现的经验法则、技巧或判断准则 |
| `ai_action` | agent 做过的关键操作，如检索、修改、验证、生成产物 |

provenance 标注规则：
- `user`: 用户明确选择、否定、确认、提供证据或提出判断
- `ai`: AI 建议、推断、执行、验证或总结出的事件
- 不确定来源时，写清依据；不要把 AI 建议伪装成用户决策

```yaml
### 研究探索记录
session_date: YYYY-MM-DD
events:
  - type: decision|experiment|dead_end|pivot|claim|heuristic|ai_action
    description: 一句话描述
    provenance: user|ai
```

## 用户可见检查点

### 检查点 1：必须说清这轮到底做了什么

不能只说“已完成”，要能让用户一眼知道：
- 做了什么
- 解决了什么

### 检查点 2：必须说清现在怎么验证

至少要告诉用户：
- 现在哪个文件、页面或结果可以直接看
- 哪个命令、页面操作或检查路径能验证
- 哪些验证已经跑过，哪些还没跑

**出口断言(任务层验证,收尾前必过)**：六段的"现在用户能做什么/怎么验"必须填**真实证据**(跑过的命令输出、文件确认、页面/远端状态),不是设想。硬断言:声明"已完成/已生效"的,必须真取证过;没跑就如实写"未验证+为什么",绝不把推断当事实。详见全局 `~/.claude/CLAUDE.md`「验证层」。

### 检查点 3：写记忆前必须确认

无论任务大小，用户没确认前都不写记忆。

### 检查点 4：必须说清剩余风险

如果还有未跑测试、未实机验证、未部署、未确认权限、未覆盖边界，要写在“未完成内容”或“下一步决策”里。

### 检查点 5：不得借收尾扩需求

下一步只能给和本轮目标直接相邻的决策，不主动开大蓝图。

## 交付证据规则

按任务类型选择证据：

| 类型 | 最低证据 |
|---|---|
| 文件/文档 | 可点击文件路径，说明新增或修改了什么 |
| 代码 | 测试/构建/lint 结果；没跑就说明原因 |
| UI | 页面 URL 或截图检查；没跑浏览器就说明未验证 |
| 远端/部署 | 服务状态、URL、日志或 hash；不能只说“已部署” |
| 调研/分析 | 产物路径、核心结论、来源边界 |
| skill 改造 | 修改文件、快照/回滚点、评分或人工检查结果 |
| 中途阻塞 | 已尝试动作、失败证据、下一步需要的输入 |

不要伪造证据。证据不足时，明确说“当前只完成到 X，Y 尚未验证”。

## 记忆更新建议

输出后明确询问：
> "建议更新 timeline 和 current-position，确认写入？"

用户明确回复"是/Y/确认"→写入。沉默/忽略/说"否"→不写入。

### 任务级记忆（tasks/ 体系，若工作区已启用）

如果本轮在某个跨会话任务上推进（工作区有 `.claude/memory/tasks/<任务>/`），收尾时**额外**提议更新这四样（治"机制在不执行"）：

1. **state.md** — 阶段 / 下一步是否变了
2. **unit-log.md** — 追加一行本轮成果（append，不覆盖）
3. **artifacts.md** — 本轮有新产出 → 登记坐标（是什么/存哪/含跨设备路径/怎么取回）
4. **INDEX.md** — 任务阶段列是否要更新

询问口径合并：
> "建议更新 timeline + tasks/<任务>/(state+log+artifacts) + INDEX，确认写入？"

仍遵守"确认才写"：用户点头才写，沉默/否 → 不写。产出物存储遵循 `~/shared-skills/STORAGE-SPEC.md`。

## 经验提炼（lessons）— Reflector

收尾时除了流水账，还要做一步**批判性提炼**：把本轮真正的复发经验沉淀成 `lessons.md` 的一条。这是经验库自进化的来源。两个触发时机：

**时机 A — 踩坑必提（强）**：用户重跑 / 纠正 / 说"不行" / 否掉方向时，必问一句：
> "这次的坑是**结构性的**（换个会话还会再踩），还是**一次性的**（网络抽风、手滑）？"

只有结构性的才提议成 learning。一次性事故不记——记噪音比不记更坏。

**时机 B — 收尾轻扫（弱）**：每次 6 段总结后，自检一句"本轮有没有产生值得记的复发经验？"
- 有 → 提议 1 条（最多 2 条）。
- 无 → 明说"本轮无新经验"。**不强凑**。

提炼出的 learning 必须遵守：

1. **写成条件句，不写绝对句**：`当 X → 做 Y`，不是 `永远做 Y`。绝对句会过拟合这一个案例，下次场景一变就误导。
2. **带 meta 行**（给 `scripts/lessons_gc.py` 体检用）：
   ```
   ## YYYY-MM-DD: 当 X → 做 Y
   <!-- meta: id=Lxxx | scope=skill名,场景标签 | applied=0 | helpful=0 | harmful=0 | last=YYYY-MM-DD -->
   ```
3. **防污染**：根因不确定时，标 `置信度=低` 并写清依据，不把猜测伪装成定论。
4. **仍是草稿**：按铁律问"确认写入 lessons.md？"，用户点头才写。沉默/否 → 不写。

## 经验召回（lessons）— 渐进披露

提炼是写入，召回是读回——闭环的另一半。skill 执行**前**，按当前任务的 scope 标签从 `lessons.md` grep 出相关条目，当侧载上下文：

```bash
grep -B1 -A6 "scope=.*<当前skill或场景>" .claude/memory/lessons.md
```

规则：
- **按 scope 过滤，不整篇塞**（防 context 膨胀；ACE 实测整篇重写会塌缩退化）。
- 命中后在回复里点一句"历史教训（自动加载）：…"，让用户知道这条经验在起作用。
- 应用了某条 → 收尾时该条 meta 的 `applied +1`；明显帮到 `helpful +1`，帮倒忙 `harmful +1`。
- 这是协议，不强制所有 skill 立刻接；谁需要谁引用（output-layer 的 `docs/style-adapt-spec.md` 是现成范例）。

## 轻量清退

### 定位

- `closeout` 可以对 `current-position` 提出清退建议
- 目标不是删历史，而是让"当前焦点/最近待办/阻塞项"保持新鲜
- **只建议，不自动处理**

### 只处理什么

- `current-position.md` 里的已完成待办
- `current-position.md` 里的已失焦事项
- `current-position.md` 里的长时间阻塞事项

### 明确不处理什么

- 不删除 `timeline.md` 历史记录
- 不删除 `decisions.md` 决策记录
- 不删除 `lessons.md` 复盘记录
- 不做自动归档系统

### 输出方式

如果检测到 `current-position` 有明显过期内容，在 6 段总结后追加：

```markdown
### 清退建议
- 可移除: [已完成事项]
- 可降级: [已失焦但以后可能还会做的事项]
- 可标暂停: [阻塞过久或等待外部输入的事项]
```

### 候选规则

- **可移除：** 本轮已完成，且下轮不再是当前焦点
- **可降级：** 不是当前优先级，但保留为 backlog 更合适
- **可标暂停：** 依赖外部决策/权限/信息，短期内无法推进

### 确认口径

有清退建议时，收尾后的询问改为：
> "建议更新 timeline 和 current-position；另外检测到 3 项可清退/降级，确认一并处理吗？"

用户明确回复"是/Y/确认"→按建议写入 `current-position`

用户回复"只写入，不清退"→只更新正常收尾内容，不处理候选清退项

用户沉默/忽略/说"否"→不写入

### timeline 建议内容

```markdown
## YYYY-MM-DD HH:MM
- **做了什么：** [简述]
- **产出：** [产出物]
- **位置：** [路径]
- **下一步：** [待办]
- **阻塞：** [无 / 原因]
```

### current-position 建议内容（轻量扁平）

```markdown
## 当前焦点
- **正在进行：** [1 句话]
- **优先级：** P0/P1/P2

## 最近待办
1. [事项]
2. [事项]

## 阻塞项
- 无 / [原因]

## 上次更新
YYYY-MM-DD HH:MM
```

### lessons 建议内容

```markdown
## YYYY-MM-DD: [标题]
**场景：** [什么情况下]
**观察：** [发生了什么]
**做法：** [怎么处理的]
**效果：** [结果如何]
**置信度：** 高/中/低
**建议：** [下次怎么做]
```

### decisions 建议内容

```markdown
## YYYY-MM-DD: [标题]
**决策：** [内容]
**背景：** [为什么]
**影响：** [影响什么]
**状态：** 已确认 / 待验证
```

## 不同状态处理

### 正常完成

- 输出 6 段总结
- 说明验证结果和产物路径
- 如有需要，追加"清退建议"
- 建议更新记忆，等确认

### 用户叫停

- 输出"任务已暂停"
- 列出已完成/未完成
- 建议更新 current-position 为"暂停"
- timeline 建议记录"[暂停] 原计划 X，已完成 Y，剩余 Z"
- 如旧待办已失效，可一并给出清退建议

### 遇到阻塞

- 输出"遇到阻塞"
- 描述问题、已尝试方案
- 建议更新 current-position 阻塞项
- 建议下一步决策
- 对已经不再适合作为当前待办的旧事项，可建议降级或暂停

### 验证失败

- 输出"验证未通过"
- 写清失败命令、失败页面、失败文件或失败状态
- 说明影响范围
- 给出回退、重试或等待用户确认的下一步
- 不把失败任务写成已完成

## 边界与异常处理

收尾本身也会遇到异常。这些情况下 closeout 不硬走流程，明说现状并降级：

| 情况 | 条件 | 怎么处理 | 对用户怎么说 |
|------|------|----------|--------------|
| 上下文不足 | 会话太短/被压缩，看不清这轮做了什么 | 不编 6 段，只写能确认的部分，其余标"未捕获" | "本轮上下文不足，只能确认 X，其余请你补充" |
| 记忆文件缺失 | `.claude/memory/` 或目标文件不存在 | 不创建整套结构，提示用户先 init | "未发现记忆文件，需先初始化工作区再写入" |
| 脚本不可用 | `lessons_gc.py` 等不存在或报错 | 跳过该步，不阻断收尾 | "经验体检脚本不可用，已跳过，收尾照常" |
| 无新内容可记 | 本轮无产出、无经验、无决策 | 不强凑记忆建议 | "本轮无值得落盘的新内容，不建议写记忆" |
| 用户已离开 | 叫停后无回应 | 只输出总结，不写任何记忆 | （静默等待，不写入） |

原则：**收尾失败要可见**——宁可说"这步没做成"，也不假装收干净了。

## 最小验收

这轮至少要做到：

1. 6 段总结完整
2. 用户现在能做什么写清楚
3. 产物路径、页面、命令结果或未验证原因写清楚
4. 剩余风险和未完成内容写清楚
5. 如有必要，给出轻量清退建议
6. 写记忆前明确询问
7. 不借收尾顺手扩新需求
8. 锚点扫描完成（见下方"锚点池摄入"），有产出则跑索引刷新

## 锚点池摄入(L0)

### 定位

任务结束时,识别本次做事过程中自动产生的"锚点"(anchor)——**做事过程中自动留下的、未来可复用的片段**(不是总结,不是反思)。

写入 `<task_dir>/anchors-YYYY-MM-DD.md`,给 `content work` 工作区在写文章时消费。

详见 `$ANCHOR_POOL_DIR/README.md`（需配置 ANCHOR_POOL_DIR 环境变量，见本 skill 的 README.md）。

### 什么算锚点

| kind | 例 |
|---|---|
| `metaphor` | 跨学科类比(扳手原型、机械自由度、苹果香蕉、agent 名动词) |
| `contrast` | 大家都说 A 但其实 B |
| `debunk` | 被吹爆的方法在 X 场景下不行 |
| `element_meta` | 元评论(我一直在混淆 X 和 Y,你注意没注意) |
| `承认无知` | 我尚不知道什么叫 dns |
| `现场临机` | 黑客松落单 → 主动找皓洁组队 |
| `长程缠斗` | 撞墙不放弃,1 小时拍板 |
| `fun_anchor` | 让用户笑出来的瞬间 |

参考样本:`$ANCHOR_POOL_DIR/seeds/wechat_writing_anchors-2026-05-25.md`

### 工作方式

1. 6 段总结输出后,做一次"锚点扫描":回看本轮过程,列出真实出现的锚点
2. **不强求**:0 条也允许,不要为了凑而凑
3. **不替用户编**:原话 / 原过程 / 真实代码片段优先
4. **格式**:每条带 `kind` / 引文 / 来源(任务路径)
5. 询问用户是否产出 `anchors-<date>.md` 到当前任务目录
6. 用户确认 → 写入 + 跑 `bash $ANCHOR_POOL_DIR/update-index.sh` 刷新跨工作区索引
7. 不确认 → 不写,锚点直接丢弃(不留在记忆里堵塞)

### 输出格式

锚点扫描在 6 段总结之后、"清退建议"之前给出。**必须严格按 `$ANCHOR_POOL_DIR/ANCHOR_TEMPLATE.md` 的 schema**——下游 stats / spark / connect 依赖此格式解析,自由格式不算合法锚点。

候选清单先以简表形式呈现给用户:

```markdown
### 锚点扫描

候选锚点(本轮产出,N 条):

| # | id | kind | 短标题 | quote 摘要 | source |
|---|---|---|---|---|---|
```

### Schema 强制项(产出时必查)

写入 `anchors-<date>.md` 前确认:

- ✅ frontmatter 含 `task / task_path / date / workspace`
- ✅ 每条锚点有 `id`(格式 `YYYYMMDD-DW|SR|CW-NN`,文件内唯一)
- ✅ 每条锚点有 `kind`(从固定 10 种里选)、`quote`、`source`
- ✅ `kind` 不在 10 种之内 → **不许产出**,要么改 kind 要么丢这条
- ✅ `quote` 是原话或具体场景,不是抽象总结
- ✅ `source` 不留空(`conversation` / `commit:xxx` / `file:xxx` 任选)
- ✅ 单条 quote 长度 ≤ 200 字,超了拆条
- ✅ 产出后跑 `bash $ANCHOR_POOL_DIR/validate.sh <文件路径>` 自检

10 种合法 `kind`:`metaphor / contrast / debunk / reclassify / inversion / element_meta / 承认无知 / 现场临机 / 长程缠斗 / fun_anchor`

### 询问口径合并

有锚点 + 有清退建议时,合并问一次:

> "建议更新 timeline + current-position;另产出 anchors-<date>.md(N 条)+ 清退 M 项,确认?"

用户回复 "是/Y/确认" → 全部写入 + 跑索引
回复 "只写记忆" → 不产 anchors,不清退
回复 "只产锚点" → 只写 anchors,不动记忆
沉默 / 否 → 都不做

## Ideas 回看(发酵期检查)

6 段总结之后 + 锚点扫描之前,顺便跑一下 ideas 回看:

```bash
bash $ANCHOR_POOL_DIR/review-ideas.sh
```

如果输出里有 🔔 或 🩶 项,简洁告知用户,不要展开:

> "顺便:`ideas.md` 里有 N 条该看了 — [想法标题],想现在过一下还是先放着?"

用户回复:
- "过一下" → 把这条调出来,做快速对撞（类似 dream-catcher 的碰撞风格：给 3 个质疑点，问用户还想不想做）
- "放着" → 不动
- "弃了" → 帮用户改 ideas.md 状态字段为 🪦
- 沉默 → 不做

**不要每次 closeout 都打断**——只在 review-ideas.sh 真的有输出时提一次。每条想法在 3 天发酵期内提醒最多 3 次(每天 1 次),超过 3 次还不动就自动进 🩶 灰。

## 沟通切片(疲惫信号触发)

closeout 不是切片的主要触发点(那个由日常对话识别),但 closeout 时如果发现这轮对话出现:
- 用户连续 5 轮以上"嗯/好/可以"型短回应
- 用户没有提任何新决策

可以在 6 段总结之后插一句:

> "这轮你回得比较短,要不要跳出来看看自己的工作状态?"

用户说"好" → 可以调出 observations.md（如果工作区配置了此文件）一起过。
用户说"不用" → 继续走完 closeout,不勉强。

**说明**：此机制源自"对话切片本身有清醒作用"的原则——工具是手段不是目的；想法要撞不要归档；对话切片能帮用户意识到自己的状态。
