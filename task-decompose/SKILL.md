# Task Decompose

## 定位

把已经分析清楚的任务拆成可执行步骤。

它不负责重新定义目标，只负责回答一件事：这件事最省力、最可验收的走法是什么。

拆解的结果必须能交给另一个 agent 或下一轮对话直接执行，不需要重新判断方向。

## Outcome Contract

- **产出：** 最多 5 个可执行子任务，每步都有目标、输入、输出、验收、风险和依赖。
- **完成标准：** 用户能看懂先做哪一步、每步做完看到什么、哪里必须拍板、失败怎么退回。
- **证据：** 来自已确认目标、当前工作区规则、用户约束、已有文件路径、可运行命令或明确的待确认项。
- **输出：** 任务拆解表和执行建议，不输出完整方案长文。

## 它在防什么坑

- 任务范围明明不大，却被拆成一大串空步骤
- 一步能做完的事，被过度项目化
- 步骤很多，但每一步都不知道验什么
- 需要用户拍板的地方被悄悄跳过
- 上一步还没产出可见结果，就依赖下一步才有价值
- 把调研、实现、验证、收尾混成一个大任务
- 没写风险和回退，失败后只能重来

## 什么时候用

- 用户确认“开始”“拆解”
- 分析结果显示需多步
- 用户显式要求拆解

## 不什么时候用

- 一步能直接做完
- 目标还没分析清楚
- 当前争议点还很多，应该先回到 `task-analyze`
- 用户只要方向判断，不要执行步骤
- 用户还没确认危险操作、外部服务、技术选型或长期方向
- 用户要的是完整项目方案，应转 `task-crafter`
- 用户要的是收尾总结，应转 `closeout`

## 默认工作方式

先收最小闭环，不为拆而拆。

默认只拆到：
- 每步都能验收
- 总步数不超过 5
- 当前会话推进不超过 3 步
- 每步都能独立产生用户可见进展
- 每个确认点都在执行前暴露

## 拆解原则

### 1. 步数

- 拆解：最多 **5** 个子任务
- 执行：每会话推进不超 **3** 步
- 超 5 步 → 建议用户收敛

### 2. 每步可验收

格式："做完后用户能看到：[具体效果]"

每步都必须有 `Done when`：

- 看到哪个文件、页面、命令结果或文档
- 哪个状态表示通过
- 哪个状态表示阻塞

### 3. 依赖清晰

```text
[步骤1] → [步骤2] → [步骤3]
```

不要设计“必须全部完成才有用”的阶段。每一步结束后，系统都应该处在一个可理解、可停下、可继续的状态。

### 4. 人决策点

以下必须插入用户确认：
- 技术选型
- 删除/覆盖现有文件
- 新增功能（非修复）
- 外部服务/API
- 全局配置、安装、部署、发布
- 涉及账号、密钥、隐私、商业数据
- 会影响多个项目目录或长期记忆规则

### 5. 风险和回退

每个子任务至少标一个风险或写“无明显风险”。

回退必须具体：

- 文件类任务：快照、备份、版本化输出或不覆盖
- 代码类任务：测试失败时回到上一可用状态
- 文档类任务：保留旧稿或生成新文件
- 外部状态任务：未确认前不执行

### 6. 范围闸门

遇到以下情况，不继续拆大计划，先收敛：

- 子任务超过 5 个
- 当前会话需要推进超过 3 步
- 需要引入新系统、新服务或复杂架构
- 依赖用户还没确认的方向
- 验收方式说不清

处理方式：输出“建议先收最小闭环”，并给出缩小后的 1-3 步版本。

## 用户可见检查点

### 检查点 1：先判断值不值得拆

如果一步能做完，就直接说不拆。

### 检查点 2：每步必须可验收

每个子任务都要能回答：
- 做完后用户能看到什么

### 检查点 3：必须标出确认点

凡是会影响范围、结构或外部依赖的地方，必须明确标注“这里要用户确认”

### 检查点 4：必须标出风险和回退

每步都要能回答：
- 可能在哪里失败
- 失败后如何回到可继续状态

### 检查点 5：不得伪装成拆解

如果只是“一步做完”的任务，直接说“不需要拆”，然后给出单步执行建议。

## 输出格式

```markdown
## 任务拆解：[名称]

### 总览
- 预计步骤：N 步（最多 5）
- 预计时间：[粗略]
- 需确认点：[列出]
- 最小闭环：[做到哪一步就已经有用户可见结果]

### 步骤 1：[名称]
- **目标：** [做什么]
- **输入：** [前置条件]
- **输出：** [产出]
- **验收：** [验证方式]
- **Done when：** [完成标准]
- **风险：** [可能问题 / 无明显风险]
- **回退：** [失败处理]
- **依赖：** 无 / 步骤 X

### 执行建议
- 顺序：[线性/并行]
- 本轮建议推进：[最多 3 步]
- 暂不做：[明确排除项]
```

简单任务输出可以压缩为：

```markdown
这件事不需要拆。

- **单步目标：** [目标]
- **验收：** [用户能看到什么]
- **风险/确认：** [如有]
- **建议执行：** [下一步]
```

## 最小验收

这轮至少要做到：

1. 先判断这事是否真的需要拆
2. 步数不超过 5
3. 每步都有目标、输入、输出、验收
4. 每步都有 Done when、风险和回退
5. 用户拍板点有明确标记
6. 说明最小闭环做到哪一步
7. 不把简单任务拆成空流程
